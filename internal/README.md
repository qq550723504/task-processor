# internal 目录

## 用途

存放项目的私有代码，这些代码不应该被外部项目导入。Go 编译器会强制执行这一规则。

## 目录结构

```
internal/
├── app/         # 应用层：业务逻辑、服务编排
├── core/        # 核心层：配置、日志、生命周期等基础功能
├── crawler/     # 爬虫层：各平台爬虫实现
├── domain/      # 领域层：领域模型、业务规则
├── infra/       # 基础设施层：认证、消息队列、锁等
├── pipeline/    # 管道层：责任链模式的处理管道
├── pkg/         # 内部工具包：可复用的工具函数
└── platforms/   # 平台层：各电商平台的业务实现
```

## 架构分层说明

### 1. core（核心层）
- 最底层，被所有其他层依赖
- 提供配置管理、日志、错误处理等基础功能
- 不依赖任何其他层

### 2. domain（领域层）
- 领域模型和业务规则
- 不依赖基础设施实现
- 定义接口，由其他层实现

### 3. infra（基础设施层）
- 实现技术细节：数据库、消息队列、HTTP 客户端等
- 实现 domain 层定义的接口
- 依赖 core 层

### 4. crawler（爬虫层）
- 各平台的爬虫实现
- 依赖 core 和 infra 层

### 5. pipeline（管道层）
- 责任链模式的处理管道
- 用于组织复杂的业务流程
- 依赖 core 层

### 6. platforms（平台层）
- 各电商平台的业务逻辑
- 依赖 crawler、domain、infra 层

### 7. app（应用层）
- 最上层，编排各个组件
- 提供对外的服务接口
- 依赖所有其他层

### 8. pkg（工具包层）
- 可复用的工具函数和类型
- 可以被任何层使用

## 依赖关系图

```
app (应用层)
 ↓
platforms (平台层)
 ↓
crawler (爬虫层) + pipeline (管道层)
 ↓
domain (领域层) + infra (基础设施层)
 ↓
core (核心层) + pkg (工具包层)
```

## 编码规范

1. 遵循依赖倒置原则：高层模块不依赖低层模块，都依赖抽象
2. 每层只能依赖下层，不能反向依赖
3. 使用接口定义层之间的契约
4. 避免循环依赖

## 注意事项

- internal 目录下的代码不能被外部项目导入
- 保持清晰的分层架构
- 每层有明确的职责边界
- 使用依赖注入管理组件依赖
