## 架构评估总结（task-processor）

### 1. 总体评价

- **架构风格**：分层架构 + DDD + 消息驱动，整体设计合理、清晰。
- **分层结构**：
  - `cmd`：各可执行程序入口。
  - `internal/app`：应用层（messaging / bootstrap / scheduler / service 等），负责业务流程编排。
  - `internal/domain`：领域层（task / product / queue / errors / model 等），负责业务规则与领域模型。
  - `internal/infra`：基础设施层（rabbitmq / http / openai / repo / di / lock / monitoring / productcrawler 等），负责技术实现。
  - `internal/core`：核心通用组件（config / lifecycle / logger / errors 等）。
  - `internal/crawler`：平台无关或偏平台的爬虫实现（amazon / alibaba1688 等）。
  - `internal/platforms`：平台层（temu / shein / amazon 等），封装各平台业务逻辑。
- **文档与重构支持**：
  - `docs/architecture`、`docs/functions`、`docs/理想架构设计方案.md`、`docs/代码拆分方案.md` 非常完备，为后续演进提供了清晰路线。
  - RabbitMQ、Bootstrap、Domain 已有较系统的重构记录与指南。

整体来看，当前项目架构成熟度较高，具备良好的演进基础。

---

### 2. 发现的主要问题与偏差

#### 2.1 分层依赖方向的例外（需要优先修正）

- **问题**：`internal/domain/product/factory/ProductServiceFactory` 位于 `domain` 层，却直接依赖：
  - `internal/infra/productcrawler`（基础设施实现）
  - 已经迁移出的原 `domain/product/repo/impl` 实现包
- **影响**：
  - 违反“领域层不依赖基础设施层”的分层原则。
  - 使 domain 层难以在测试/重用时保持纯净，也为后续服务拆分和依赖注入带来额外耦合。

#### 2.2 应用层某些组件过于“胖”（ServiceManager / RabbitMQService）

- 文档中自己已标记的高优先级问题：
  - `app/messaging` 中的 `ServiceManager`、`RabbitMQService` 职责过多：
    - 同时承担连接管理、消费者管理、任务编排、平台注册等职责。
- **影响**：
  - 难以按平台/用例粒度拆分服务（如只做 temu-listing 或 amazon-crawler）。
  - 可测试性和可维护性变差，变更时波及面大。

#### 2.3 平台层与爬虫层的重复逻辑

- 已在 `docs/functions/README.md` 中标记为中优先级：
  - 平台处理器接口不统一，各平台实现差异较大。
  - `crawler` 目录中存在平台间可共用的逻辑（浏览器池、页面操作、通用提取器等）尚未完全抽象。

#### 2.4 Bootstrap / DI 使用字符串 key（可改善）

- `internal/app/bootstrap/component_adapters.go` 中：
  - 通过 `"temuProcessor"`、`"sheinProcessor"`、`"schedulerService"` 等字符串从 DI 容器获取实例。
- **影响**：
  - 类型安全和 IDE 支持略差。
  - 随着组件增多，字符串 key 出错难以及早发现。

### 3. 优化建议与演进方向

#### 3.1 修正 domain 对 infra 的反向依赖（高优先级）

- **目标**：domain 层只定义业务接口与模型，不直接引用 infra 实现。
- **具体建议**：
  - 将 `ProductServiceFactory` 从 `internal/domain/product/factory` 移出 domain 层，例如：
    - `internal/app/product/factory`（应用层负责组合 domain 与 infra）；或
    - `internal/infra/factory/product_service_factory.go`（专门的工厂/装配层，由 app 调用）。
  - domain 层只保留：
    - `domain/product/service`：`ProductService` 等领域服务。
    - `domain/product/repo`：`CrawlerRepository`、`CacheRepository` 等接口。
  - 由 app/bootstrap 或 DI 容器在启动时组装：
    - 从 infra 创建 `CrawlerRepositoryImpl`、`CacheRepositoryImpl` 等实现。
    - 传入 `ProductService` 构造函数。

#### 3.2 拆分 app/messaging 的“胖管理器”（高优先级）

- **目标**：让应用层服务按用例/职责拆分，方便服务级扩展与部署。
- **拆分思路**：
  - 拆出按功能划分的应用服务：
    - `TaskSubmissionService`：只负责将任务发布到 RabbitMQ。
    - `TaskConsumptionService`：只负责消费任务队列并分发给平台处理器。
    - `ResultReportingService`：只负责将处理结果上报管理系统。
  - 将 `RabbitMQService` 简化为对 `infra/rabbitmq.Client` 的门面：
    - 对上只暴露领域无关的接口，如 `PublishTaskMessage`、`SubscribeTaskQueue`。
    - 业务规则与任务结构放在 domain，应用层负责调用门面，把消息映射为领域对象/DTO。

这样，后续按 `docs/代码拆分方案.md` 把不同平台/能力拆成独立服务时，只需选择性地组合这些应用服务即可。

#### 3.3 平台处理器与爬虫模块的抽象统一（中优先级）

- **平台处理器**：
  - 定义统一接口（示例）：
    - `type PlatformProcessor interface { ProcessTask(ctx context.Context, task *Task) error }`
  - 各平台实现各自的 `TemuProcessor`、`SheinProcessor`、`AmazonProcessor`，但对上暴露统一接口，方便：
    - 应用层通用注册与路由。
    - 未来拆分独立服务时重用代码。
- **爬虫模块**：
  - 在 `internal/crawler` 下引入公共子包，例如 `crawler/core` 或 `crawler/common`：
    - 浏览器池管理。
    - 通用的页面操作与选择器封装。
    - 通用反爬策略、验证码处理框架。
  - 平台特定包（`amazon`、`alibaba1688` 等）只保留差异化逻辑（选择器、结构化解析规则、平台特有流程）。

#### 3.4 改善 Bootstrap / DI 的类型安全（中优先级）

- **目标**：减少字符串 key 带来的隐患，提高可维护性。
- **建议**：
  - 在 `bootstrap` 或 `infra/di` 上增加轻量包装：
    - 用类型化构造方法，而不是在业务代码中大量硬编码字符串 key。
  - 逐步用“小模块构造函数”替代“全局容器 + 字符串 key”：
    - 例如 `NewTemuProcessor(logger, cfg, deps...)`、`NewSchedulerService(...)`。
  - 保留 DI 容器用于“装配图管理”，减少在业务入口随意 `Get("xxx")`。

### 4. 与“理想架构 / 拆分方案”的差距与路线

- **目标架构**（见 `docs/理想架构设计方案.md`、`docs/代码拆分方案.md`）：
  - K8s + 微服务 + Serverless 浏览器池。
  - 按业务划分独立服务（crawler / listing / operation / infra service）。
  - 按平台进一步拆分（`amazon-crawler`、`1688-crawler`、`temu-listing`、`shein-listing`、`pricing-service`、`inventory-service` 等）。
- **当前状态**：
  - 代码和文档已经为多服务入口与容器化提供了清晰蓝图（`cmd/README.md`、拆分方案示例 main.go / Makefile / Dockerfile / K8s YAML）。
  - 核心阻力主要在于：
    - 某些应用层组件职责过大、对多个平台/能力耦合过深。
    - 少数跨层依赖（如 `ProductServiceFactory`）未完全按照分层原则收口。
- **推荐演进顺序**：
  1. **修复分层依赖问题**：迁移 `ProductServiceFactory`；确认 domain 不再 import infra。
  2. **拆 `app/messaging` 的大管理器**：按用例拆服务，收窄公开 API，便于复用。
  3. **抽象平台处理器接口 & 爬虫公共模块**：为平台/服务粒度拆分打下抽象基础。
  4. **按照拆分方案逐步增加独立 `cmd/*` 入口与 Docker/K8s 部署**：从负载最重、收益最高的服务开始（如 `amazon-crawler`、`temu-listing`）。

---

### 5. 优先级总览

- **高优先级**
  - 将 `ProductServiceFactory` 从 `domain` 迁出，消除 domain 对 infra 的直接依赖。
  - 拆分 `app/messaging` 中的 `ServiceManager`、`RabbitMQService`，按职责/用例重组为多个小型应用服务。
- **中优先级**
  - 为各平台处理器定义统一接口，抽象平台无关逻辑。
  - 在 `crawler` 中提取公共组件，减少跨平台重复。
  - 渐进式提升 Bootstrap / DI 的类型安全性，弱化字符串 key 使用。
- **长期目标**
  - 按文档中的微服务与容器化方案拆分、部署服务，逐步向“理想架构”演进，同时配套监控与成本优化。

### 6. 已实施的高优先级重构（2026-03）

- **6.1 修正 domain 对 infra 的反向依赖**
  - 已将 `ProductServiceFactory` 从 `internal/domain/product/factory` 迁移到应用层 `internal/app/service/product_factory.go`。
  - `domain/product` 目录现在只包含领域模型、仓储接口和领域服务，不再直接依赖 `infra/productcrawler` 等基础设施实现。
  - 具体效果：依赖方向收敛为 `app/service → domain + infra`，满足“领域层不依赖基础设施层”的分层原则。

- **6.2 为消息消费端引入应用层抽象，并瘦身 ServiceManager**
  - 在 `internal/app/service` 新增 `MessagingService` 抽象和 `NewMessagingService` 工厂，当前由 `messaging.ServiceManager` 实现。
  - 在 `internal/app/messaging` 中新增 `HTTPServerManager`，负责 `/health`、`/ready`、`/metrics`、`/stats` 的 HTTP 服务启动与停止：
    - 健康检查和指标逻辑从 `ServiceManager` 中抽离，`ServiceManager` 仅持有并在启动/停止时委托给该组件。
  - 引入 `ShutdownCoordinator`，将 **信号监听（SIGINT/SIGTERM）与优雅关闭** 从 `ServiceManager` 中抽离，统一在组件内实现；`ServiceManager` 仅负责组合和委托。
  - 入口程序收敛：`cmd/temu-listing`、`cmd/shein-listing`、`cmd/amazon-crawler`、`cmd/1688-crawler` 统一通过 `app/service.NewMessagingService` 创建消息服务，并在启动后使用 `serviceManager.Wait()` 阻塞等待，由内部组件触发优雅关闭（入口侧不再重复实现信号处理与 Stop 调用）。
  - 对外行为（端口、路由、队列消费与处理流程）保持不变，仅内部职责拆分更清晰，为后续继续拆分 `ServiceManager` / `RabbitMQService` 做好铺垫。

- **6.3 TEMU 试点服务拆分现状（可复制模板）**
  - **当前启动链路（简化）**：
    - `cmd/temu-listing/main.go`
      - 加载配置 → `app/service.NewMessagingService(...)` → 注册 `TEMU` 处理器 → `Start()` → `Wait()`
    - 内部：`ServiceManager` 负责组合 `RabbitMQService`（消费/重连/队列初始化）、`HTTPServerManager`（健康检查/metrics）、`ResultReporter`、`LoadMonitor`，并由 `ShutdownCoordinator` 统一监听信号并触发优雅关闭。
  - **为什么这条链路适合作为试点**：
    - 入口职责极简（仅配置 + 组装 + 启动 + 等待），方便直接复制到其它平台入口。
    - 信号处理/关闭流程统一在内部组件，避免每个 `cmd/*` 重复实现且不一致。
  - **下一步拆分到“独立可部署服务”的关键检查项**：
    - **队列边界**：确认仅消费 TEMU 相关队列（routing key / queue name 与 registry 注册一致），避免隐式消费其它平台消息。
    - **依赖边界**：TEMU 处理器不应直接 import 其它平台包（SHEIN/Amazon），共用逻辑下沉到 `platforms/common` 或 domain/app 层接口。
    - **配置与端口**：健康检查/metrics 端口通过配置可覆盖，避免多服务并行部署时冲突。
