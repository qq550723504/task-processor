# Task Processor 架构重构计划

## 📋 重构目标

基于之前的架构分析，本次重构旨在解决以下核心架构问题，同时**完全保留现有业务逻辑**：

1. **依赖注入容器职责过重** - 重构为职责单一的容器
2. **全局状态滥用** - 消除全局变量，改为依赖注入
3. **生命周期管理混乱** - 统一组件生命周期接口
4. **配置管理耦合** - 解耦配置加载和业务逻辑
5. **错误处理过度设计** - 简化错误处理机制

## 🎯 重构原则

### ✅ 保留内容
- **所有业务逻辑**：TEMU、SHEIN处理器的完整功能
- **所有配置项**：现有配置结构和参数
- **所有API接口**：对外暴露的服务接口
- **所有监控指标**：现有的监控和健康检查
- **所有调度任务**：核价、同步、活动报名等任务

### 🔄 重构内容
- **架构层面**：依赖注入、生命周期管理、配置管理
- **代码组织**：文件结构、职责分离、接口设计
- **错误处理**：简化错误类型和处理流程

## 📊 重构阶段规划

### 第一阶段：基础架构重构（核心）

#### 1.1 依赖注入容器重构
**目标**：将现有的臃肿容器拆分为职责单一的组件

**现状问题**：
```go
// 当前容器承担太多职责
type Container struct {
    // 服务管理 + 配置管理 + 认证管理 + 资源管理
    configService    *service.ConfigService
    authService      *service.AuthService
    config           *config.Config
    amazonProcessor  *amazon.AmazonProcessor
    // ... 更多职责
}
```

**重构方案**：
```go
// 1. 纯粹的依赖注入容器
type DIContainer interface {
    Register(name string, factory Factory) error
    RegisterSingleton(name string, factory Factory) error
    Get(name string) (interface{}, error)
    Close() error
}

// 2. 专门的应用启动器
type ApplicationBootstrap struct {
    container DIContainer
    configManager ConfigManager
    lifecycleManager LifecycleManager
}

// 3. 专门的配置管理器
type ConfigManager interface {
    Load(source ConfigSource) (*Config, error)
    Validate(cfg *Config) error
    Watch(callback func(*Config)) error
}
```

**实施步骤**：
1. 创建新的DIContainer接口和实现
2. 创建ApplicationBootstrap来管理应用启动
3. 保持现有Container作为LegacyContainer，逐步迁移
4. 所有业务逻辑通过工厂函数注册到新容器

#### 1.2 消除全局状态
**目标**：移除所有全局变量，改为依赖注入

**现状问题**：
```go
// 全局状态导致测试困难和并发不安全
var globalConfig *config.Config
var sharedAmazonProcessor *amazon.AmazonProcessor
var sharedManagementClient *management.ClientManager

func GetGlobalConfig() *config.Config
func SetGlobalConfig(cfg *config.Config)
func GetSharedAmazonProcessor() *amazon.AmazonProcessor
```

**重构方案**：
```go
// 通过依赖注入传递所有依赖
type ProcessorService struct {
    config           *config.Config          // 注入
    amazonProcessor  *amazon.AmazonProcessor // 注入
    managementClient *management.ClientManager // 注入
}

// 工厂函数负责创建和注入依赖
func NewProcessorService(
    cfg *config.Config,
    amazonProcessor *amazon.AmazonProcessor,
    managementClient *management.ClientManager,
) ProcessorService {
    return &processorServiceImpl{
        config: cfg,
        amazonProcessor: amazonProcessor,
        managementClient: managementClient,
    }
}
```

**实施步骤**：
1. 识别所有全局变量和函数
2. 为每个使用全局状态的组件添加依赖注入
3. 通过容器管理所有共享资源
4. 保持业务逻辑完全不变

#### 1.3 统一生命周期管理
**目标**：为所有组件提供统一的生命周期接口

**现状问题**：
- 组件启动停止逻辑分散
- 没有统一的依赖关系管理
- 错误恢复机制不完善

**重构方案**：
```go
// 统一的组件接口
type Component interface {
    Name() string
    Dependencies() []string
    Priority() int
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
    IsRunning() bool
}

// 生命周期管理器
type LifecycleManager interface {
    Register(component Component) error
    StartAll(ctx context.Context) error
    StopAll(ctx context.Context) error
    GetStatus() map[string]ComponentStatus
}
```

**实施步骤**：
1. 为现有服务创建Component适配器
2. 保持现有启动停止逻辑不变
3. 通过适配器模式包装现有组件
4. 逐步迁移到新的生命周期管理

### 第二阶段：配置和错误处理优化

#### 2.1 配置管理解耦
**目标**：将配置管理从业务逻辑中分离

**重构方案**：
```go
// 配置源抽象
type ConfigSource interface {
    Read() ([]byte, error)
    Watch(ctx context.Context, callback func([]byte)) error
    Name() string
}

// 配置管理器
type ConfigManager interface {
    Load(source ConfigSource) (*Config, error)
    Validate(cfg *Config) error
    GetCurrent() *Config
    Reload() error
}
```

#### 2.2 简化错误处理
**目标**：简化过度设计的错误系统

**现状问题**：
```go
// 过度复杂的错误类型
type AppError struct {
    Code    string
    Message string
    Details map[string]interface{}
    File    string
    Line    int
}
```

**重构方案**：
```go
// 使用标准错误处理 + 错误分类
var (
    ErrConfigInvalid = errors.New("配置无效")
    ErrServiceUnavailable = errors.New("服务不可用")
    ErrAuthFailed = errors.New("认证失败")
)

// 错误包装提供上下文
func ProcessTask(task *Task) error {
    if err := validate(task); err != nil {
        return fmt.Errorf("任务验证失败: %w", err)
    }
    return nil
}
```

### 第三阶段：组件适配和迁移

#### 3.1 业务组件适配
**目标**：为现有业务组件创建新架构适配器

**保持不变的组件**：
- `TemuProcessor` - 完整保留所有TEMU处理逻辑
- `SheinProcessor` - 完整保留所有SHEIN处理逻辑
- `AmazonProcessor` - 完整保留所有Amazon爬虫逻辑
- `TaskFetcher` - 完整保留任务获取逻辑
- `SchedulerService` - 完整保留所有调度任务

**适配器示例**：
```go
// 为现有处理器创建组件适配器
type TemuProcessorComponent struct {
    *lifecycle.BaseComponent
    processor *temu.TemuProcessor
    config    *config.Config
    // 保持所有原有依赖
}

func (c *TemuProcessorComponent) Start(ctx context.Context) error {
    // 调用原有的启动逻辑，不做任何修改
    return c.processor.Start(ctx)
}

func (c *TemuProcessorComponent) Stop(ctx context.Context) error {
    // 调用原有的停止逻辑，不做任何修改
    c.processor.Close(ctx)
    return nil
}
```

#### 3.2 服务注册迁移
**目标**：将所有服务迁移到新的依赖注入容器

**迁移策略**：
```go
// 注册所有现有服务到新容器
func RegisterServices(container DIContainer, cfg *config.Config) error {
    // 注册配置
    container.RegisterSingleton("config", func(c DIContainer) (interface{}, error) {
        return cfg, nil
    })
    
    // 注册Amazon处理器（保持原有创建逻辑）
    container.RegisterSingleton("amazonProcessor", func(c DIContainer) (interface{}, error) {
        config := c.Get("config").(*config.Config)
        return amazon.NewAmazonProcessor(&config.Amazon), nil
    })
    
    // 注册TEMU处理器（保持原有创建逻辑）
    container.Register("temuProcessor", func(c DIContainer) (interface{}, error) {
        config := c.Get("config").(*config.Config)
        amazonProcessor := c.Get("amazonProcessor").(*amazon.AmazonProcessor)
        managementClient := c.Get("managementClient").(*management.ClientManager)
        
        // 使用原有的构造函数，不修改任何业务逻辑
        return temu.NewTemuProcessor(ctx, config, logger, managementClient, amazonProcessor), nil
    })
    
    // ... 注册所有其他服务
}
```
