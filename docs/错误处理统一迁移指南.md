## 概述

### 目标
统一项目中的错误处理机制，提高代码可维护性和可靠性。

### 当前问题
1. 错误处理方式不统一（`fmt.Errorf`, `errors.New`, 自定义错误类型）
2. 部分代码使用 `panic` 和 `log.Fatal`，导致程序崩溃
3. 缺少统一的错误分类和重试机制
4. 错误信息不够结构化，难以追踪和分析

### 解决方案
使用统一的 `internal/core/errors` 包，提供：
- 标准化的错误码和错误类型
- 错误包装和上下文信息
- 重试机制和错误处理器
- 安全的panic恢复

## 迁移步骤

### 步骤1: 替换标准错误创建

#### 之前
```go
// 使用 errors.New
return errors.New("产品不存在")

// 使用 fmt.Errorf
return fmt.Errorf("获取产品失败: %w", err)
```

#### 之后
```go
import "task-processor/internal/core/errors"

// 创建新错误
return errors.New(errors.ErrCodeValidation, "产品不存在")

// 包装现有错误
return errors.Wrap(err, errors.ErrCodeExternalAPI, "获取产品失败")

// 格式化错误消息
return errors.Wrapf(err, errors.ErrCodeExternalAPI, "获取产品失败: productID=%s", productID)
```

### 步骤2: 替换 panic 和 log.Fatal

#### 之前
```go
// 在构造函数中使用 panic
func NewProcessor(config *Config) *Processor {
    if config == nil {
        panic("config不能为空")
    }
    return &Processor{config: config}
}

// 在main函数中使用 log.Fatal
if err := app.Start(); err != nil {
    log.Fatalf("启动失败: %v", err)
}
```

#### 之后
```go
// 构造函数返回错误
func NewProcessor(config *Config) (*Processor, error) {
    if config == nil {
        return nil, errors.New(errors.ErrCodeConfig, "config不能为空")
    }
    return &Processor{config: config}, nil
}

// main函数中使用 logger.Fatal（仅在main函数中）
if err := app.Start(); err != nil {
    logger.Fatalf("启动失败: %v", err) // 仅在main函数中使用
}

// 其他地方返回错误
func (s *Service) Start() error {
    if err := s.init(); err != nil {
        return errors.Wrap(err, errors.ErrCodeSystem, "初始化失败")
    }
    return nil
}
```

### 步骤3: 使用错误处理器

#### 之前
```go
func (p *Processor) Process(task Task) error {
    err := p.doSomething()
    if err != nil {
        logger.Errorf("处理失败: %v", err)
        return err
    }
    return nil
}
```

#### 之后
```go
func (p *Processor) Process(task Task) error {
    err := p.doSomething()
    if err != nil {
        // 使用错误处理器统一处理
        return p.errorHandler.Handle(err)
    }
    return nil
}

// 初始化时创建错误处理器
func NewProcessor(logger *logrus.Logger) *Processor {
    return &Processor{
        errorHandler: errors.NewDefaultErrorHandler(logger),
    }
}
```

### 步骤4: 实现重试机制

#### 之前
```go
func (s *Service) FetchData() error {
    var lastErr error
    for i := 0; i < 3; i++ {
        err := s.fetch()
        if err == nil {
            return nil
        }
        lastErr = err
        time.Sleep(time.Second * time.Duration(i+1))
    }
    return lastErr
}
```

#### 之后
```go
func (s *Service) FetchData(ctx context.Context) error {
    config := errors.DefaultRetryConfig()
    config.MaxRetries = 3
    
    return errors.Retry(ctx, config, func() error {
        return s.fetch()
    })
}
```

### 步骤5: 安全的goroutine

#### 之前
```go
go func() {
    // 可能panic的代码
    result := riskyOperation()
    process(result)
}()
```

#### 之后
```go
errors.SafeGo(func() {
    // panic会被自动捕获和记录
    result := riskyOperation()
    process(result)
}, logger)
```

## 使用示例

### 示例1: 基本错误处理

```go
package service

import (
    "task-processor/internal/core/errors"
)

type ProductService struct {
    logger errors.Logger
}

func (s *ProductService) GetProduct(id string) (*Product, error) {
    if id == "" {
        return nil, errors.New(errors.ErrCodeValidation, "产品ID不能为空")
    }
    
    product, err := s.repository.FindByID(id)
    if err != nil {
        return nil, errors.Wrap(err, errors.ErrCodeExternalAPI, "查询产品失败")
    }
    
    if product == nil {
        return nil, errors.New(errors.ErrCodeTaskNotFound, "产品不存在")
    }
    
    return product, nil
}
```

### 示例2: 带重试的网络请求

```go
func (s *APIClient) Request(ctx context.Context, url string) (*Response, error) {
    config := errors.DefaultRetryConfig()
    config.MaxRetries = 5
    config.InitialDelay = 2 * time.Second
    
    var response *Response
    err := errors.Retry(ctx, config, func() error {
        resp, err := s.httpClient.Get(url)
        if err != nil {
            return errors.Wrap(err, errors.ErrCodeNetwork, "HTTP请求失败")
        }
        response = resp
        return nil
    })
    
    return response, err
}
```

### 示例3: 错误处理器

```go
type TaskProcessor struct {
    errorHandler errors.ErrorHandler
    logger       *logrus.Logger
}

func NewTaskProcessor(logger *logrus.Logger) *TaskProcessor {
    return &TaskProcessor{
        errorHandler: errors.NewDefaultErrorHandler(logger),
        logger:       logger,
    }
}

func (p *TaskProcessor) ProcessTask(task Task) error {
    err := p.process(task)
    if err != nil {
        // 统一处理错误
        handledErr := p.errorHandler.Handle(err)
        
        // 判断是否需要重试
        if p.errorHandler.ShouldRetry(handledErr) {
            return p.retryTask(task)
        }
        
        // 判断是否需要终止
        if p.errorHandler.ShouldTerminate(handledErr) {
            p.logger.Error("遇到关键错误，停止处理")
            return handledErr
        }
        
        return handledErr
    }
    return nil
}
```

### 示例4: 多错误合并

```go
func (s *Service) BatchProcess(items []Item) error {
    var errs []error
    
    for _, item := range items {
        if err := s.processItem(item); err != nil {
            errs = append(errs, err)
        }
    }
    
    // 合并所有错误
    return errors.Combine(errs...)
}
```

### 示例5: 安全执行

```go
func (s *Service) RiskyOperation() error {
    return errors.SafeExecute(func() error {
        // 可能panic的代码
        result := s.doSomethingRisky()
        return s.processResult(result)
    }, s.logger)
}
```

## 最佳实践

### 1. 错误码选择

- 使用最具体的错误码
- 网络相关错误使用 `ErrCodeNetwork`
- 验证错误使用 `ErrCodeValidation`
- 外部API错误使用对应的API错误码

### 2. 错误消息

- 消息应该清晰描述问题
- 包含必要的上下文信息
- 避免暴露敏感信息
- 使用中文或英文保持一致

```go
// 好的错误消息
errors.Newf(errors.ErrCodeValidation, "产品ID格式无效: %s", productID)

// 不好的错误消息
errors.New(errors.ErrCodeValidation, "错误")
```

### 3. 错误包装

- 始终包装底层错误，保留错误链
- 添加有意义的上下文信息
- 使用 `%w` 格式化动词

```go
// 正确
return errors.Wrap(err, errors.ErrCodeExternalAPI, "调用Amazon API失败")

// 错误 - 丢失了原始错误
return errors.New(errors.ErrCodeExternalAPI, "调用Amazon API失败")
```

### 4. 日志记录

- 使用错误处理器统一记录
- 关键错误使用 Error 级别
- 可重试错误使用 Warn 级别
- 避免重复记录同一个错误

```go
// 好的做法
err := p.errorHandler.Handle(err)
return err

// 不好的做法 - 重复记录
logger.Errorf("错误: %v", err)
return p.errorHandler.Handle(err)
```

### 5. Panic 使用

- 仅在初始化阶段使用 `Must` 和 `MustValue`
- 运行时错误应该返回 error
- 使用 `SafeGo` 启动 goroutine

```go
// 初始化阶段可以使用
func init() {
    config := errors.MustValue(loadConfig())
}

// 运行时应该返回错误
func (s *Service) Process() error {
    if err := s.validate(); err != nil {
        return err // 不要panic
    }
    return nil
}
```

### 6. 重试策略

- 只重试可恢复的错误
- 设置合理的重试次数和延迟
- 使用指数退避避免雪崩
- 支持context取消

```go
config := errors.DefaultRetryConfig()
config.MaxRetries = 3
config.InitialDelay = time.Second
config.MaxDelay = 30 * time.Second
config.BackoffFactor = 2.0
```

## 常见问题

### Q1: 什么时候使用 `log.Fatal`？

**A:** 仅在 `main` 函数中使用，用于处理无法恢复的启动错误。其他地方应该返回 error。

```go
// main.go - 可以使用
func main() {
    if err := app.Start(); err != nil {
        logger.Fatalf("启动失败: %v", err)
    }
}

// service.go - 不要使用
func (s *Service) Start() error {
    if err := s.init(); err != nil {
        return err // 返回错误，不要Fatal
    }
    return nil
}
```

### Q2: 如何处理第三方库的错误？

**A:** 使用 `errors.Wrap` 包装第三方错误，添加上下文信息。

```go
resp, err := http.Get(url)
if err != nil {
    return errors.Wrap(err, errors.ErrCodeNetwork, "HTTP请求失败")
}
```

### Q3: 是否需要迁移所有现有代码？

**A:** 建议分阶段迁移：
1. 优先迁移新代码
2. 修改现有代码时顺便迁移
3. 关键路径优先迁移
4. 逐步替换旧的错误处理

### Q4: 如何判断错误是否可重试？

**A:** 使用 `errors.IsRetryable(err)` 或错误处理器的 `ShouldRetry` 方法。

```go
if errors.IsRetryable(err) {
    // 可以重试
}

// 或使用错误处理器
if handler.ShouldRetry(err) {
    // 可以重试
}
```

### Q5: 如何添加自定义错误码？

**A:** 在 `internal/core/errors/errors.go` 中添加新的错误码常量。

```go
const (
    // 现有错误码...
    
    // 新增错误码
    ErrCodeCustom ErrorCode = "CUSTOM_ERROR"
)
```

### Q6: 错误处理会影响性能吗？

**A:** 影响极小：
- 错误创建：纳秒级
- 错误包装：纳秒级
- 堆栈信息：微秒级（可选）
- 对正常流程无影响
