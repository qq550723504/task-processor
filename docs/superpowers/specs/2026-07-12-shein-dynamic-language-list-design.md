# SHEIN 动态商品语言列表设计

## 背景

当前产品名和 SKC 名均通过 `submitprep.GetTargetLanguagesByRegion` 的硬编码地区映射决定目标语言。SHEIN 商品发布页实际通过以下接口返回当前店铺可用语言：

`POST /spmp-api-prefix/spmp/basic/get_language_list`

请求体为 `{}`，响应的 `info.data` 包含 `language_abbr`、`language_name` 和 `input_mode`。硬编码映射可能缺少接口新增语言，也可能生成当前店铺不支持的语言。

## 目标

- 产品名和 SKC 名使用 SHEIN 接口返回的同一目标语言列表。
- 每个发布任务最多请求一次语言列表。
- 接口失败或没有有效语言时继续发布，并回退现有地区映射。
- 保持上一轮按分类、语言动态限制名称长度的行为。

## 非目标

- 不改变语言检测算法。
- 不改变翻译 API 或 AI 优化策略。
- 不新增跨任务持久缓存。
- 不移除 `GetTargetLanguagesByRegion`；它继续作为兼容回退。
- 不改变产品描述的生成与长度限制。

## API 层

在现有 SHEIN product API 客户端中增加语言列表查询：

- endpoint 常量及 getter；
- `LanguageListItem` 响应模型；
- `QueryLanguageList()` 客户端方法；
- 使用现有 `APIRequest` 和业务响应码校验。

接口请求固定为 `POST`，请求体为空 JSON 对象。

## 语言列表规范化

增加聚焦的语言列表规范化逻辑：

- 仅保留 `input_mode > 0` 的条目；
- `language_abbr` 去除首尾空白并转为小写；
- 忽略空语言代码；
- 按规范化后的语言代码去重；
- 保持接口首次出现顺序。

接口返回的第一项有效语言作为主展示语言，其他语言按接口顺序生成多语言内容。

## 任务级加载与数据流

1. 分类选择及产品数据初始化完成。
2. 翻译处理器发现任务上下文尚未初始化目标语言时，调用语言列表接口。
3. 成功响应经规范化后保存到 `TaskContext.TargetLanguages`。
4. 若请求失败或规范化结果为空，记录告警，并将当前地区映射结果保存到上下文。
5. 产品名生成直接使用 `TaskContext.TargetLanguages`。
6. SKC runtime 从同一任务上下文复制目标语言列表，生成相同语言集合的 SKC 名。
7. 每个语言继续使用任务级名称长度配置进行 Unicode 字符截断。

非空切片表示语言加载已经完成，避免产品和 SKC 阶段重复请求。保存到上下文及复制到 SKC runtime 时复制切片，避免调用方意外修改共享底层数组。

## 错误处理

- 网络错误、非成功业务码或响应解析失败：告警并回退地区映射。
- `info.data` 为空或所有条目被过滤：告警并回退地区映射。
- 某条语言无效：忽略该条，不影响其他有效语言。
- 回退映射意外为空：最终使用 `en`，保证至少生成一个名称。

语言列表错误不使发布任务失败。

## 测试策略

按测试驱动方式覆盖：

- API 使用 `POST`、正确路径和 `{}` 请求体，并解析响应；
- `input_mode <= 0`、空语言代码被过滤；
- 大小写与空白规范化、去重和顺序保持；
- 翻译处理阶段每个任务只请求一次；
- 产品名使用动态语言列表；
- SKC 名使用与产品名相同的任务级列表；
- 接口失败和空有效响应均回退地区映射；
- 动态语言仍应用对应的动态名称长度配置。

## 验收标准

- 示例响应仅包含有效的 `en`、`fr` 时，产品名和每个 SKC 名均生成且仅生成英文、法文内容。
- 同一任务语言接口请求次数为一次。
- 接口不可用时任务继续执行，并保持原地区语言行为。
- 完整 SHEIN 测试与相关包静态检查通过。
