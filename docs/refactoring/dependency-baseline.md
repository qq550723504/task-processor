# Dependency Baseline

This document records the dependency and package-shape baseline used for project-wide refactoring.

It should be generated before broad package moves and updated only when intentionally refreshing the baseline.

## 1. How to Generate

From the repository root:

```powershell
./scripts/analyze-project-deps.ps1 | Tee-Object -FilePath docs/refactoring/dependency-baseline-output.txt
```

For stricter validation:

```powershell
./scripts/analyze-project-deps.ps1 -FailOnViolation
```

At the start of the refactoring program, `-FailOnViolation` may fail because legacy exceptions may already exist. In that case:

1. Run the advisory command without `-FailOnViolation`.
2. Copy the results into this document.
3. Classify each violation as either:
   - known legacy exception,
   - must-fix before migration,
   - false positive,
   - rule needs refinement.
4. Only then promote the script to CI enforcement.

## 11. Baseline Refresh Policy

Refresh this baseline when:

- A major package move is completed.
- A new package boundary is enforced in CI.
- A marketplace package is normalized to the target architecture.
- The root `internal/listingkit` file count drops significantly.

Do not refresh this baseline merely to hide regressions.
