# 跨境电商智能上架系统 - 理想架构设计方案

生成时间: 2026-03-10

## 🎯 架构设计目标

### 核心目标
1. **高资源利用率**：充分利用计算资源，避免浪费
2. **弹性伸缩**：根据业务负载自动扩缩容
3. **高可用性**：单点故障不影响整体服务
4. **易扩展**：支持新平台、新功能快速接入
5. **成本优化**：按需使用资源，降低运营成本

### 业务需求
- 管理 300+ 店铺（未来可能增长到 1000+）
- 支持多源平台爬取（Amazon、1688、沃尔玛、速卖通等）
- 支持多目标平台上架（SHEIN、TEMU、Amazon 等）
- 智能运营（自动核价、活动报名、库存监测）
- 高并发处理（单日处理数万个商品）

## 🔑 核心设计理念

### 1. 服务拆分（微服务化）

**按业务功能拆分**：
```
爬虫服务 (Crawler Service)
├── Amazon 爬虫
├── 1688 爬虫
├── 沃尔玛爬虫
└── 速卖通爬虫

上架服务 (Listing Service)
├── TEMU 上架
├── SHEIN 上架
└── Amazon 上架

运营服务 (Operation Service)
├── 自动核价服务
├── 活动报名服务
└── 库存监测服务

基础服务 (Infrastructure Service)
├── 浏览器池服务
├── 图片处理服务
└── AI 服务代理
```

**优势**：
- 独立部署、独立扩展
- 故障隔离，不互相影响
- 技术栈可以不同
- 团队可以并行开发

### 2. 容器化 + Kubernetes

**为什么选择 K8s**：
- ✅ 自动扩缩容（HPA）
- ✅ 自动故障恢复
- ✅ 负载均衡
- ✅ 滚动更新
- ✅ 资源限制和配额
- ✅ 服务发现

**部署配置示例**：
```yaml
# 爬虫服务部署
apiVersion: apps/v1
kind: Deployment
metadata:
  name: amazon-crawler
spec:
  replicas: 10  # 初始 10 个 Pod
  selector:
    matchLabels:
      app: amazon-crawler
  template:
    metadata:
      labels:
        app: amazon-crawler
    spec:
      containers:
      - name: crawler
        image: task-processor/amazon-crawler:latest
        resources:
          requests:
            cpu: "1"
            memory: "2Gi"
          limits:
            cpu: "2"
            memory: "4Gi"
        env:
        - name: RABBITMQ_URL
          valueFrom:
            secretKeyRef:
              name: rabbitmq-secret
              key: url
---
# 自动扩缩容配置
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: amazon-crawler-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: amazon-crawler
  minReplicas: 5      # 最少 5 个 Pod
  maxReplicas: 100    # 最多 100 个 Pod
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU 70% 时扩容
  - type: External
    external:
      metric:
        name: rabbitmq_queue_messages_ready
        selector:
          matchLabels:
            queue: amazon.crawler
      target:
        type: AverageValue
        averageValue: "30"  # 队列积压 > 30 时扩容
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # 扩容稳定窗口 1 分钟
      policies:
      - type: Percent
        value: 50        # 每次扩容 50%
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容稳定窗口 5 分钟
      policies:
      - type: Pods
        value: 2         # 每次缩容 2 个 Pod
        periodSeconds: 60
```

### 3. Serverless 浏览器池

**为什么用 Serverless**：
- 浏览器是重资源（每个 500MB-1GB 内存）
- 使用频率不均匀（有高峰和低谷）
- 按需创建，用完销毁，成本最优

**方案选择**：

**方案 A：AWS Lambda + Puppeteer**
```javascript
// Lambda 函数
exports.handler = async (event) => {
    const puppeteer = require('puppeteer-core');
    const chromium = require('chrome-aws-lambda');
    
    const browser = await puppeteer.launch({
        args: chromium.args,
        executablePath: await chromium.executablePath,
        headless: true,
    });
    
    const page = await browser.newPage();
    await page.goto(event.url);
    
    // 执行爬取逻辑
    const data = await page.evaluate(() => {
        // 提取数据
    });
    
    await browser.close();
    
    return {
        statusCode: 200,
        body: JSON.stringify(data)
    };
};
```

**方案 B：阿里云函数计算 + Headless Chrome**
```go
// Go Worker 调用 Serverless 浏览器
func (c *Crawler) FetchProduct(url string) (*Product, error) {
    // 调用 Serverless 函数
    resp, err := c.invokeServerlessFunction(map[string]interface{}{
        "url": url,
        "action": "fetch_product",
    })
    
    if err != nil {
        return nil, err
    }
    
    var product Product
    json.Unmarshal(resp, &product)
    return &product, nil
}
```

**方案 C：自建浏览器池服务（推荐）**
```go
// 浏览器池服务（独立部署）
type BrowserPoolService struct {
    pool *BrowserPool
}

// HTTP API
func (s *BrowserPoolService) HandleRequest(w http.ResponseWriter, r *http.Request) {
    var req BrowserRequest
    json.NewDecoder(r.Body).Decode(&req)
    
    // 从池中获取浏览器
    browser := s.pool.Acquire()
    defer s.pool.Release(browser)
    
    // 执行任务
    result := browser.Execute(req)
    
    json.NewEncoder(w).Encode(result)
}

// K8s 部署配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: browser-pool
spec:
  replicas: 3  # 初始 3 个 Pod
  template:
    spec:
      containers:
      - name: browser-pool
        image: browser-pool:latest
        resources:
          requests:
            memory: "8Gi"  # 每个 Pod 8GB 内存
          limits:
            memory: "16Gi"
        env:
        - name: POOL_SIZE
          value: "10"  # 每个 Pod 10 个浏览器
---
# 自动扩缩容
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: browser-pool-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: browser-pool
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

**成本对比**：
| 方案 | 成本 | 优点 | 缺点 |
|------|------|------|------|
| AWS Lambda | 按调用次数 | 完全 Serverless | 冷启动慢，成本可能高 |
| 阿里云函数计算 | 按调用次数 | 国内访问快 | 同上 |
| 自建浏览器池 | 固定成本 | 性能好，可控 | 需要运维 |

**推荐**：自建浏览器池服务（K8s 部署 + HPA）

### 4. 分布式任务调度

**为什么需要调度器**：
- 任务分片（大任务拆分成小任务）
- 负载均衡（任务均匀分配）
- 故障转移（节点故障自动重试）
- 优先级调度（紧急任务优先处理）

**方案选择**：

**方案 A：XXL-Job（推荐）**
```java
// Java 后端集成 XXL-Job
@XxlJob("createListingTasks")
public void createListingTasks() {
    // 获取所有待上架商品
    List<Product> products = productService.getPendingProducts();
    
    // 分片参数
    int shardIndex = XxlJobHelper.getShardIndex();
    int shardTotal = XxlJobHelper.getShardTotal();
    
    // 分片处理
    for (int i = 0; i < products.size(); i++) {
        if (i % shardTotal == shardIndex) {
            Product product = products.get(i);
            // 创建任务并发送到 RabbitMQ
            taskService.createTask(product);
        }
    }
}
```

**方案 B：自研调度器**
```go
// 分布式任务调度器
type TaskScheduler struct {
    redis    *redis.Client
    rabbitmq *RabbitMQ
}

// 任务分片
func (s *TaskScheduler) ScheduleTask(task *Task) error {
    // 1. 任务分片
    shards := s.shardTask(task)
    
    // 2. 负载均衡
    for _, shard := range shards {
        worker := s.selectWorker(shard)
        s.assignTask(worker, shard)
    }
    
    return nil
}

// 选择最优 Worker
func (s *TaskScheduler) selectWorker(task *TaskShard) string {
    // 从 Redis 获取所有 Worker 的负载信息
    workers := s.getWorkerMetrics()
    
    // 计算负载分数
    var bestWorker string
    var lowestScore float64 = math.MaxFloat64
    
    for _, worker := range workers {
        score := s.calculateScore(worker, task)
        if score < lowestScore {
            lowestScore = score
            bestWorker = worker.ID
        }
    }
    
    return bestWorker
}
```

**推荐**：XXL-Job（成熟稳定，功能完善）

// Go Worker 调用图片处理服务
func (s *ListingService) ProcessImages(images []string) ([]string, error) {
    var processedImages []string
    
    for _, imageURL := range images {
        // 调用图片处理服务
        resp, err := s.imageService.Process(ImageRequest{
            URL: imageURL,
            Operations: []Operation{
                {Type: "compress", Quality: 85},
                {Type: "resize", Width: 1000, Height: 1000},
                {Type: "watermark", Text: "SHEIN"},
            },
        })
        
        if err != nil {
            return nil, err
        }
        
        processedImages = append(processedImages, resp.URL)
    }
    
    return processedImages, nil
}

### 成本估算

**云服务器成本**（阿里云/腾讯云）：
- 10 台 8核16G：¥1,000/台/月 × 10 = ¥10,000/月
- 年成本：¥120,000

**中间件成本**：
- RabbitMQ 集群：¥2,000/月
- Redis 集群：¥1,500/月
- MySQL 集群：¥2,000/月
- 对象存储：¥500/月
- 年成本：¥72,000

**运维成本**：
- 监控告警：¥1,000/月
- 日志分析：¥1,000/月
- 年成本：¥24,000

**总成本**：¥216,000/年

**对比云电脑方案**：
- 云电脑方案：100 台 × ¥500/月 = ¥600,000/年
- K8s 方案：¥216,000/年
- **节省**：¥384,000/年（64% 成本节省）
