## 1. 当前问题分析

### 1.1 目录结构问题

#### 问题 A：`internal/common` 与 `internal/platforms` 职责重叠

**现状：**
```
internal/common/
├── amazon/          # Amazon 相关公共代码
├── shein/           # Shein 相关公共代码
├── temu/            # Temu 相关公共代码
└── ...

internal/platforms/
├── amazon/          # Amazon 平台实现
├── shein/           # Shein 平台实现
└── temu/            # Temu 平台实现
```

**问题：** 同一平台的代码分散在两个目录，增加了理解和维护成本。开发者需要在两个目录间跳转才能理解完整的平台逻辑。

---

#### 问题 B：`internal/scheduler/scheduler` 冗余嵌套

**现状：**
```
internal/scheduler/
├── scheduler/                    # 冗余嵌套！
│   ├── common.go
│   ├── monitor_scheduler.go
│   └── sync_scheduler.go
├── monitor_scheduler.go          # 与子目录文件名重复
├── safe_scheduler.go
└── sync_scheduler.go             # 与子目录文件名重复
```

**问题：** 目录嵌套冗余，且存在同名文件，容易造成混淆。

---

#### 问题 C：`internal/memory` 与 `internal/common/memory` 重复

**现状：**
```
internal/memory/
├── cookie_manager.go
├── daily_count_manager.go
├── manager.go
├── relisting_queue_manager.go
└── shop_pause_manager.go

internal/common/memory/
├── cookie_manager.go             # 同名文件！
├── daily_count_manager.go        # 同名文件！
├── manager.go                    # 同名文件！
├── relisting_queue_manager.go    # 同名文件！
└── shop_pause_manager.go         # 同名文件！
```

**问题：** 两个目录包含完全相同的文件名，极可能是重复代码或遗留代码。

---

#### 问题 D：`internal` 目录过于扁平

**现状：** `internal/` 下有 24 个子目录，缺乏层次结构：

```
internal/
├── api/           ├── goroutine/    ├── pipeline/     ├── task/
├── auth/          ├── lifecycle/    ├── platforms/    ├── types/
├── bootstrap/     ├── logger/       ├── repo/         ├── updater/
├── clients/       ├── memory/       ├── scheduler/    ├── utils/
├── common/        ├── middleware/   ├── service/      ├── worker/
├── config/        ├── model/        
├── container/     ├── monitoring/   
├── errors/        
```

**问题：**
- 难以快速理解项目架构
- `utils/` 和 `common/utils/` 职责重叠
- `types/` 和 `model/` 职责相似
- `errors/` 只有一个文件，过度拆分

---

#### 问题 E：配置与数据混合

**现状：**
```
config/
├── config-dev.yaml              # 应用配置
├── sensitive_words.json         # 业务数据
├── sensitive_words_shein.json   # 业务数据
└── sensitive_words_temu.json    # 业务数据
```

**问题：** 应用配置和业务数据文件混在一起，职责不清。

#### 问题 F：二进制文件管理

- `dist/` 目录包含多个版本的编译产物（虽已在 .gitignore 中）
- `chrome/` 目录包含浏览器二进制（虽已在 .gitignore 中）

**问题：** 这些目录存在于工作区，可能会被误提交。建议在 README 中说明如何获取这些依赖。

## 5. 风险控制

### 5.1 重构原则

1. **小步快跑**：每次只做一个小改动，确保可以编译通过
2. **频繁提交**：每完成一个小步骤就提交，便于回滚
3. **保持可运行**：每次提交后确保程序可以正常运行
4. **先测试后重构**：对关键模块先补充测试

### 5.2 每个阶段的检查清单

#### 阶段完成检查

- [ ] 所有代码可以编译通过 (`go build ./...`)
- [ ] 所有测试通过 (`go test ./...`)
- [ ] 程序可以正常启动
- [ ] 核心功能可以正常工作
- [ ] 没有循环依赖 (`go mod tidy` 无报错)

### 5.3 回滚策略

每个阶段开始前创建 Git 分支：

```bash
# 阶段 1 开始
git checkout -b refactor/phase-1-cleanup

# 阶段 1 完成后合并
git checkout main
git merge refactor/phase-1-cleanup

# 阶段 2 开始
git checkout -b refactor/phase-2-merge-duplicates
```

### 5.4 Import 路径更新工具

推荐使用 `goimports` 或 IDE 的重构功能来批量更新 import 路径：

```bash
# 使用 sed 批量替换（Linux/Mac）
find . -name "*.go" -exec sed -i 's|task-processor/internal/utils|task-processor/internal/pkg/utils|g' {} \;

# 或使用 gorename 工具
go install golang.org/x/tools/cmd/gorename@latest
```

### 5.5 依赖关系检查

重构后检查是否有循环依赖：

```bash
go mod tidy
go build ./...
```

如果出现循环依赖，需要：
1. 提取公共接口到独立包
2. 使用依赖注入解耦
3. 重新考虑包的划分

### B. 预计工时

| 阶段 | 预计工时 | 风险等级 |
|------|----------|----------|
| 阶段 1：清理整理 | 1-2 周 | 低 |
| 阶段 2：合并重复 | 2-3 周 | 中 |
| 阶段 3：重组结构 | 2-3 周 | 中 |

### C. 建议的执行顺序

1. 先完成阶段 1，稳定后再继续
2. 阶段 2 和 3 可以部分并行
3. 阶段 4 风险最高，建议在其他阶段完成后再开始
4. 每个阶段完成后，建议运行完整的回归测试
