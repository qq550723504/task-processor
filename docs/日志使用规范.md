# 日志使用规范

## 1. 日志级别使用指南

### Debug
- 用途：详细的调试信息，仅在开发和问题排查时使用
- 示例：
  - 函数参数和返回值
  - 循环迭代的详细信息
  - 中间计算结果

### Info
- 用途：重要的业务流程信息
- 示例：
  - 任务开始/完成
  - 状态变更
  - 关键操作成功

### Warn
- 用途：警告信息，不影响主流程但需要关注
- 示例：
  - 重试操作
  - 使用默认值
  - 性能降级

### Error
- 用途：错误信息，影响功能但系统可继续运行
- 示例：
  - API调用失败
  - 数据验证失败
  - 可恢复的错误

### Fatal
- 用途：致命错误，系统无法继续运行
- 示例：
  - 配置文件缺失
  - 必要服务无法启动
  - 数据库连接失败

## 2. 日志记录规范

### 2.1 获取Logger实例

**推荐方式**：通过依赖注入或全局Logger

```go
// 方式1：通过依赖注入（推荐）
type MyService struct {
    logger *logrus.Entry
}

func NewMyService(logger *logrus.Entry) *MyService {
    return &MyService{
        logger: logger.WithField("component", "my_service"),
    }
}

// 方式2：使用全局Logger（适用于工具函数）
import "task-processor/internal/core/logger"

func MyUtilFunction() {
    log := logger.GetGlobalLogger("my_util")
    log.Info("执行工具函数")
}
```

**禁止方式**：
```go
// ❌ 禁止直接使用logrus全局方法
logrus.Info("message")

// ❌ 禁止使用fmt.Printf
fmt.Printf("debug: %v\n", data)

// ❌ 禁止在每个函数中创建新logger
logger := logrus.New()
```

### 2.2 结构化日志字段规范

**字段命名规则**：
- 使用 snake_case（小写+下划线）
- 保持简洁明了
- 使用统一的字段名

**标准字段**：
```go
logger.WithFields(logrus.Fields{
    // 业务标识
    "task_id":      taskID,
    "product_id":   productID,
    "tenant_id":    tenantID,
    "store_id":     storeID,
    
    // 追踪信息
    "trace_id":     traceID,
    "request_id":   requestID,
    
    // 组件信息
    "component":    "service_name",
    "platform":     "temu",
    
    // 性能指标
    "duration_ms":  duration.Milliseconds(),
    "retry_count":  retryCount,
    
    // 错误信息
    "error_code":   errorCode,
    "error_type":   errorType,
}).Info("操作完成")
```

### 2.3 日志消息规范

**消息格式**：
- 使用中文或英文，保持一致
- 简洁明了，描述清楚发生了什么
- 避免冗余信息（字段中已有的不要重复）

**好的示例**：
```go
logger.WithFields(logrus.Fields{
    "task_id": 123,
    "duration_ms": 1500,
}).Info("任务处理完成")
```

**不好的示例**：
```go
// ❌ 消息中重复字段信息
logger.WithFields(logrus.Fields{
    "task_id": 123,
}).Info("任务123处理完成")

// ❌ 消息过于冗长
logger.Info("开始处理任务，任务ID是123，产品ID是ABC，这是一个TEMU平台的任务")
```

### 2.4 错误日志规范

**使用 WithError**：
```go
if err != nil {
    logger.WithError(err).WithFields(logrus.Fields{
        "task_id": taskID,
        "operation": "fetch_product",
    }).Error("获取产品失败")
    return err
}
```

**错误日志和错误返回**：
- 在错误发生的地方记录详细日志
- 向上层返回包装后的错误
- 避免在调用链的每一层都记录相同错误

```go
// ✅ 正确：在底层记录详细日志
func (s *Service) fetchData() error {
    data, err := s.api.Get()
    if err != nil {
        s.logger.WithError(err).WithFields(logrus.Fields{
            "url": s.api.URL,
            "retry_count": s.retryCount,
        }).Error("API调用失败")
        return fmt.Errorf("获取数据失败: %w", err)
    }
    return nil
}

// ✅ 正确：上层只返回错误，不重复记录
func (h *Handler) Process() error {
    if err := h.service.fetchData(); err != nil {
        return err // 不再记录日志
    }
    return nil
}
```

### 2.5 性能优化

**延迟字符串格式化**：
```go
// ✅ 正确：使用字段，避免不必要的格式化
logger.WithFields(logrus.Fields{
    "user_id": userID,
    "count": count,
}).Debug("处理用户数据")

// ❌ 错误：即使Debug级别不输出，也会执行格式化
logger.Debugf("处理用户数据: user_id=%d, count=%d", userID, count)
```

**复用Logger实例**：
```go
// ✅ 正确：在结构体中保存带上下文的logger
type TaskHandler struct {
    logger *logrus.Entry
}

func NewTaskHandler(taskID int64) *TaskHandler {
    return &TaskHandler{
        logger: logger.GetGlobalLogger("task_handler").WithField("task_id", taskID),
    }
}

func (h *TaskHandler) Process() {
    h.logger.Info("开始处理") // 自动包含task_id
}
```

## 3. 特殊场景处理

### 3.1 测试代码

```go
func TestMyFunction(t *testing.T) {
    // 使用测试专用logger
    testLogger := logrus.New()
    testLogger.SetLevel(logrus.DebugLevel)
    testLogger.SetOutput(os.Stdout)
    
    // 或者使用全局logger
    logger.SetGlobalLogLevel("debug")
}
```

### 3.2 命令行工具

```go
func main() {
    // 初始化日志系统
    systemInit := bootstrap.NewSystemInitializer(&bootstrap.SystemConfig{
        LogConfig: &logger.LogConfig{
            Level:   "info",
            Format:  "text",  // 命令行使用text格式更易读
            Console: true,
        },
    })
    
    if err := systemInit.Initialize(); err != nil {
        log.Fatalf("系统初始化失败: %v", err)
    }
    defer systemInit.Shutdown()
}
```

### 3.3 长时间运行的任务

```go
func (s *Service) ProcessBatch(items []Item) {
    total := len(items)
    
    for i, item := range items {
        // 定期输出进度日志
        if i%100 == 0 || i == total-1 {
            s.logger.WithFields(logrus.Fields{
                "processed": i + 1,
                "total": total,
                "progress_pct": float64(i+1) / float64(total) * 100,
            }).Info("批处理进度")
        }
        
        s.processItem(item)
    }
}
```

## 4. 日志配置

### 4.1 配置文件示例

```yaml
log:
  level: info          # debug, info, warn, error, fatal
  format: json         # json, text
  output_file: logs/app.log
  max_size: 100        # MB
  console: true        # 是否同时输出到控制台
```

### 4.2 环境变量

```bash
# 开发环境
export LOG_LEVEL=debug
export LOG_FORMAT=text

# 生产环境
export LOG_LEVEL=info
export LOG_FORMAT=json
```

## 5. 迁移检查清单

从旧的日志方式迁移到新规范时，检查以下项目：

- [ ] 移除所有 `logrus.Info()` 等全局调用
- [ ] 移除所有 `fmt.Printf()` 调试输出
- [ ] 移除所有 `log.Printf()` 标准库调用
- [ ] 统一字段命名为 snake_case
- [ ] 添加必要的上下文字段（task_id, trace_id等）
- [ ] 确保错误日志使用 `WithError()`
- [ ] 检查日志级别是否合适
- [ ] 确保logger通过依赖注入传递

## 6. 常见问题

### Q: 什么时候使用Info，什么时候使用Debug？
A: Info用于记录重要的业务流程（如任务开始/完成、状态变更），Debug用于详细的调试信息（如中间变量、循环细节）。

### Q: 是否需要在每个函数入口记录日志？
A: 不需要。只在关键节点记录，避免日志过多。

### Q: 如何在日志中追踪请求链路？
A: 在context中传递trace_id和request_id，在创建logger时添加这些字段。

### Q: 生产环境应该使用什么日志级别？
A: 建议使用Info级别，Error和Warn会自动输出。

### Q: 如何临时开启Debug日志排查问题？
A: 使用 `logger.SetGlobalLogLevel("debug")` 动态调整，无需重启服务。
