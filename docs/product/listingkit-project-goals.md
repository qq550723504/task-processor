# ListingKit 项目目标与范围

## 文档状态

- 状态：Active
- 更新日期：2026-06-20
- 适用对象：产品、运营、研发、QA、项目负责人

## 1. 项目使命

ListingKit 的使命是：

> 将来自 SDS、1688、POD、海外仓或人工素材的商品信息，转化为可审核、可修复、可追踪、可提交的多平台 Listing 资料包，帮助跨境电商团队规模化完成从商品素材到平台可售商品的生产过程。

ListingKit 不是单纯的采集器、AI 文案工具或通用任务处理器。它是一套面向跨境商品上架的操作工作台和自动化生产系统。

## 2. 要解决的核心问题

### 2.1 来源商品资料不统一

不同来源提供的标题、属性、规格、变体、价格、图片和模板信息结构不同，运营需要反复人工整理，容易遗漏事实字段或引入错误。

ListingKit 需要建立平台无关的标准商品事实和资产结构，使来源变化不会直接污染平台 Listing 流程。

### 2.2 同一商品需要重复适配多个平台

Amazon、SHEIN、TEMU、Walmart 等平台对类目、属性、销售属性、图片、SKU、价格和文案的要求不同。直接复制来源资料无法稳定通过平台校验，也不能支撑差异化运营。

ListingKit 需要在标准商品事实之上，构建平台特定的 Listing 资料包和提交规则。

### 2.3 文案和图片生产成本高

商品标题、卖点、详情文案、主图、白底图、场景图、设计稿和尺寸图通常需要运营、美工和技术人员跨工具协作。

ListingKit 需要把 AI 内容生成、图片处理和商品素材组织纳入统一流程，同时保留人工审核和替换能力。

### 2.4 自动化结果不可解释、失败后难恢复

黑盒生成和一键提交在真实生产环境中风险很高。任务可能卡在来源同步、图片生成、类目匹配、属性补全、平台预校验或远端提交等任意阶段。

ListingKit 需要让用户知道：

- 当前任务处在哪个阶段；
- 为什么不能继续；
- 可以在什么位置修复；
- 哪些操作可以重试；
- 什么时候需要工程介入；
- 重试是否会产生重复商品或重复任务。

### 2.5 单任务工具无法支撑运营规模

真实运营会同时处理多个商品、店铺和批次，并面对部分成功、部分失败、部分需人工修复的混合结果。

ListingKit 需要支持批次、队列、部分成功、失败项重试、状态筛选、运营复盘和多租户数据隔离。

## 3. 目标用户

### 3.1 上架运营

上架运营负责选品、生成资料、审核图片和商品字段、修复阻断项、保存草稿或发布。

他们需要：

- 更少的重复录入；
- 清晰的下一步动作；
- 可恢复的失败流程；
- 可批量处理的工作台。

### 3.2 运营负责人

运营负责人关注产能、失败分布、发布成功率、工程介入比例和店铺配置健康。

他们需要：

- 任务和队列全局视图；
- 按平台、店铺、状态、阻断类型筛选；
- 每日失败复盘；
- 可量化的流程指标。

### 3.3 研发与 QA

研发与 QA 负责保证来源、商品事实、图片、平台规则、工作台和提交状态之间的契约一致。

他们需要：

- 明确的模块边界；
- 稳定的 API 和状态枚举；
- 真实接口验收报告；
- 可重复的失败样例和恢复测试。

## 4. 核心业务闭环

```text
来源商品 / 素材
  -> 来源身份与可用性校验
  -> 标准商品事实与资产
  -> 文案 / 图片 / 设计生产
  -> 平台 Listing 资料包
  -> readiness 检查
  -> 人工审核与修复
  -> 保存平台草稿或正式发布
  -> 状态追踪、失败恢复与运营复盘
```

对于当前重点 SDS-to-SHEIN 流程，目标链路是：

```text
SDS 选品
  -> Batch
  -> Compatibility Item
  -> Generation Attempt
  -> Materialized Design
  -> ListingKit Task
  -> SHEIN Workspace
  -> Save Draft / Publish
```

## 5. 核心产品能力

### 5.1 来源接入与标准化

- SDS 商品与设计模板信息读取；
- 1688 等来源商品采集；
- 人工文本、图片或 URL 输入；
- 标题、描述、属性、规格、价格、图片和变体标准化；
- 来源身份、版本和新鲜度记录。

### 5.2 标准商品事实与资产

- 平台无关的 canonical product；
- 商品图片、设计稿、模板、蒙版和衍生资产；
- 字段来源、置信度和需审核标记；
- 稳定缓存和动态运行时覆盖。

### 5.3 内容与图片生产

- 标题、卖点和详情文案生成；
- 平台风格差异化；
- 图片审核、抠图、白底图、场景图和压缩；
- SDS 设计稿和商品图生成；
- 按商品、颜色、尺寸或兼容表面组织素材。

### 5.4 平台适配

- 平台类目、属性和销售属性映射；
- 平台图片、SKU、变体和价格规则；
- 平台 Listing 资料包预览；
- 平台 readiness 和 blocker taxonomy。

### 5.5 审核与修复工作台

- 显示商品事实和平台资料包；
- 对类目、属性、图片、价格和 SKU 进行人工修复；
- blocker 跳转到明确修复区域；
- 历史版本、最终稿确认和修订记录。

### 5.6 提交、幂等与恢复

- 保存草稿和正式发布；
- 可观察的提交阶段；
- 幂等键和重复请求保护；
- 失败阶段、原因和可恢复性；
- 远端结果与本地状态一致性。

### 5.7 批次与运营控制台

- 多商品批次；
- item / attempt / design 级进度；
- 部分成功和失败项重试；
- 任务列表、队列和 next action；
- 设置健康检查和失败复盘。

### 5.8 多租户与权限

- 租户、用户和店铺隔离；
- 角色授权；
- 上传资产、任务、工作台和配置的租户边界；
- 所有持久化查询默认进行租户作用域校验。

## 6. 当前产品重点

当前最完整、最优先的产品闭环是 SHEIN，尤其是 SDS/POD 商品批量上架链路。

现阶段目标不是继续增加平台数量，而是先达到：

```text
运营可以独立处理真实 SHEIN 任务；
失败后可以理解、修复或安全重试；
批次中一个商品失败不会阻断其他商品；
重复操作不会创建重复任务或重复远端商品；
任务创建、草稿保存和正式发布具有准确状态语义。
```

Amazon、TEMU 和 Walmart 的后续产品化应复用 SHEIN 已验证的状态、门禁、恢复和提交模型，而不是分别建设互不一致的流程。

## 7. 产品原则

### 7.1 后端是最终业务门禁

前端校验用于改善体验，但来源可用性、baseline、店铺、商品归属、兼容性、readiness 和提交资格必须由后端最终确认。

### 7.2 自动化必须可解释

系统不能只返回成功或失败。每个关键阶段需要稳定状态、reason code、用户动作和工程诊断字段。

### 7.3 人工审核是正式流程的一部分

人工审核不是自动化失败的兜底，而是高风险字段和平台发布前的正式控制点。

### 7.4 部分成功优先于全有或全无

批次中某个候选失败时，其他独立候选应继续执行。系统必须分别返回 created、reused、rejected 和 failed。

### 7.5 状态不能夸大进度

以下状态必须严格区分：

```text
ListingKit task 已创建
task readiness 已通过
SHEIN 草稿已保存
SHEIN 商品已发布
```

不能根据标题或本地操作意图推测远端状态。

### 7.6 重试必须幂等

网络超时、用户重复点击、Worker 重启和并发恢复不得导致重复 ListingKit task、重复图片任务或重复平台商品。

### 7.7 单一事实源优先

商品事实、批次归属、提交 attempt 和远端结果必须各有清晰事实源，避免多个兼容模型同时拥有写权限。

## 8. 产品范围

### 8.1 当前范围内

- 商品来源接入和标准化；
- 商品事实、变体和图片资产；
- AI 文案和图片生产；
- SHEIN Listing 适配、审核和提交；
- 批次、队列、失败恢复和设置健康检查；
- Amazon、TEMU、Walmart 的资料包或后续平台化准备；
- 多租户、权限、审计和可观测性。

### 8.2 当前不作为主目标

- 订单履约、仓储和物流管理；
- 广告投放和站内营销自动化；
- 完整 ERP、PIM 或 OMS 替代；
- 多个不同 SDS 商品合并成一个 SHEIN Listing；
- 完整的专业平面设计软件能力；
- 在 SHEIN 主链路未稳定前同时建设多个完整平台工作台。

## 9. 关键成功指标

### 9.1 结果指标

- 保存草稿成功率；
- 正式发布成功率；
- 失败后无需工程介入恢复率；
- 重复远端提交数量；
- 批次实际创建任务数与预计候选数一致率。

### 9.2 流程指标

- 从选品到任务创建耗时；
- 从任务完成到可提交耗时；
- readiness 一次通过率；
- 每任务平均修复次数；
- 批次中部分失败后继续完成率。

### 9.3 质量指标

- 未知任务状态数量；
- 未知 blocker key 数量；
- 空错误响应数量；
- 重复任务或孤儿任务数量；
- 需要通过标题或日志猜测状态的路径数量。

## 10. 当前发布门槛

SDS-to-SHEIN 批量链路进入生产稳定阶段前，必须证明：

```text
每个 approved design x 每个 eligible selection 正确展开；
Batch 与 ListingKit task 的关系可持久化并能刷新恢复；
并发或重复请求只产生一个有效任务；
baseline、store、ownership 和 compatibility 门禁由后端执行；
共享生成按兼容表面分组；
任务创建与 SHEIN 提交状态准确区分；
已创建任务的设计不会被破坏性重生成；
批量提交在单项失败后继续执行；
真实保存草稿路径通过；
至少一个受控失败和恢复路径通过。
```
