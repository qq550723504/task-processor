# ListingKit 产品路线图

## 目的

这份路线图把 ListingKit 后续完善点按产品价值和交付风险排序。它不是研发排期，而是用于判断“下一步应该先补什么”的产品优先级基线。

优先级判断标准：

- 是否直接影响 SHEIN 主链路能否稳定完成。
- 是否减少运营人员查日志、找工程师、重复刷新和手动定位问题。
- 是否能让真实接口联调、版本验收和日常运营更可控。
- 是否为后续 TEMU、Amazon、Walmart 产品化复用能力。

## 当前阶段判断

ListingKit 已经具备从 SDS / 来源素材到 SHEIN 工作台再到保存草稿或发布的主链路。当前主要问题不是缺少单点能力，而是产品闭环还需要更稳定：

- 主链路需要真实接口验收沉淀。
- 失败恢复需要 SOP 化。
- 任务列表和队列页需要承担运营排查角色。
- 设置页需要从“填写配置”升级为“配置健康检查”。
- SHEIN 提交过程需要更强的可观测性和幂等保护。

## P0：先把 SHEIN 主链路做稳

P0 的目标是保证一个运营人员能独立完成 SHEIN 任务，不需要靠工程师解释状态或查日志。

### 1. 真实接口验收沉淀

现状：

- `web/listingkit-ui/REAL_API_VALIDATION_CHECKLIST.md` 已有字段级清单。
- 但每轮联调结果还缺固定的验收报告沉淀方式。

待完善：

- 为真实接口联调新增报告模板。
- 每轮记录 task_id、输入、状态流转、workspace payload、readiness、保存草稿、发布和失败恢复。
- 把未知状态、未知 blocker key、空错误响应作为必须关闭的问题。

验收标准：

- 至少一条真实任务从创建到发布完整通过。
- 至少一条失败路径能被用户从 UI 理解并恢复。
- 没有未知任务状态和未知 readiness blocker key。

### 2. 错误恢复 SOP

现状：

- 操作指南里已有恢复处理原则。
- 但具体错误类型还没有独立手册。

待完善：

- 补齐图片上传失败、SDS 同步失败、类目解析失败、属性缺失、workspace 缺数据、保存草稿失败、发布失败等场景。
- 每类错误明确用户可处理动作、需要工程介入的边界和记录字段。

验收标准：

- 运营看到失败后知道下一步去哪。
- QA 能按 SOP 主动触发至少一个失败样例。
- 工程收到问题时能拿到 task_id、接口响应和页面表现。

### 3. SHEIN 提交状态机

现状：

- 已有 `docs/superpowers/plans/2026-05-07-listingkit-shein-submit-state-machine.md`。
- 当前风险是提交过程只看最终结果，不容易解释“卡在哪一步”。

待完善：

- 提交 phase 可见：validate、prepare_product、upload_images、pre_validate、submit_remote、persist_result。
- 用户重复点击时具备幂等保护。
- 同任务同动作并发提交不会重复调用 SHEIN 远端接口。

验收标准：

- 保存草稿或发布进行中时，UI 能显示当前阶段。
- 失败后能看到失败阶段和原因。
- 同一 idempotency key 重放不会重复提交。

### 4. readiness blocker 稳定映射

现状：

- 前端已有 blocker key 映射基线。
- 真实接口仍可能返回新 key 或只返回 label/message。

待完善：

- 后端维护稳定 blocker taxonomy。
- 前端 unknown blocker 有明确兜底展示和记录方式。
- 联调报告强制记录未知 key。

验收标准：

- 所有阻断项都能跳到明确修复区域，或显示“暂不支持自动跳转”的可理解兜底。
- 新 blocker key 不会静默导致用户无路可走。

## P1：提升运营效率和可诊断性

P1 的目标是减少批量任务中的人工搜索、重复检查和配置排查成本。

### 1. 任务列表运营化

现状：

- 任务列表已作为入口存在。
- 产品上还需要明确它是“恢复和继续工作”的工作台，而不只是浏览页。

待完善：

- 状态、平台、更新时间、阻断类型、提交状态筛选。
- 每行给出最合适的继续入口：状态页、工作台、失败恢复或提交结果。
- 明确标识可提交、需修复、生成失败、提交失败。

验收标准：

- 运营能从列表中快速找到当天失败任务和可提交任务。
- 不需要复制 task_id 到其他页面才能继续处理。

### 2. 队列页 SOP 和信息结构

现状：

- 队列页能展示 queue、resolved action、recovery summary。
- 但用户何时使用队列页、如何解释这些信息还不够明确。

待完善：

- 定义队列页使用场景。
- 明确 Review、Retry、Inspect 的区别。
- 区分运营可处理问题和工程问题。

验收标准：

- 运营负责人能用队列页判断任务卡住原因。
- 队列页信息能支持每日失败任务复盘。

### 3. 设置页配置健康检查

现状：

- 设置页管理 AI client、SHEIN 店铺和价格规则。
- 缺少“当前配置是否可用”的诊断能力。

待完善：

- AI client 连通性检查。
- SHEIN 店铺 token、权限、类目接口检查。
- SDS 接口或登录态检查。
- 图片模型配置完整性检查。
- 显示配置影响范围。

验收标准：

- 新建任务前可以确认关键配置是否可用。
- 配置错误能在设置页暴露，而不是等任务失败后才发现。

### 4. SHEIN Studio 批量任务效率

现状：

- 已有批量任务 tracker 和 publish gate。
- 仍需要更清楚处理部分成功、部分失败、部分需修复的情况。

待完善：

- 批量任务按可提交、需修复、处理中、失败分组。
- 批量保存草稿失败时支持按失败项重试。
- 批量结果能回到任务列表继续处理。

验收标准：

- 一批任务中部分失败不会阻断其他可提交任务。
- 用户能清楚知道哪些任务需要单独进入工作台。

## P2：扩展平台产品化

P2 的目标是把 SHEIN 中已经跑通的产品经验迁移到其他平台。

### 1. TEMU 产品化定义

现状：

- TEMU 有较多架构文档和 pipeline 文档。
- 缺少类似 SHEIN 的产品流程定义。

待完善：

- 定义 TEMU 是否支持工作台审核、提交、失败恢复。
- 明确 TEMU 的类目、属性、图片和价格阻断项。
- 复用 ListingKit 任务状态和恢复模型。

### 2. Amazon ListingKit 边界

现状：

- Amazon 已有 workbench API 文档。
- ListingKit 里的 Amazon 资料包和 Amazon Listing pipeline 的产品边界需要说明。

待完善：

- 明确 Amazon ListingKit 是资料包生成还是完整工作台。
- 区分 `amazonlisting` 专用工作台和 ListingKit 多平台包。
- 统一用户入口说明。

### 3. Walmart 产品化评估

现状：

- Walmart 当前更多是资料包目标。

待完善：

- 明确是否近期进入人工审核和提交流程。
- 如果不进入，应在产品文档中标记为“资料包生成目标”，避免用户误解。

## 产品指标

后续每个优化都应尽量绑定指标。

建议先跟踪：

- 从选品到任务创建耗时。
- 从任务完成到进入工作台耗时。
- 从任务完成到可提交耗时。
- readiness 一次通过率。
- 平均每任务修复次数。
- 保存草稿成功率。
- 发布成功率。
- 失败后无需工程介入恢复率。
- 未知状态和未知 blocker key 数量。

## 推荐执行顺序

1. 固化真实接口验收报告模板。
2. 完成错误恢复手册。
3. 推进 SHEIN 提交状态机。
4. 补任务列表和队列页 SOP。
5. 补设置页健康检查设计。
6. 再启动 TEMU / Amazon / Walmart 产品化文档。

这个顺序优先解决“当前主链路是否能稳定完成”，再解决“批量运营是否高效”，最后再扩展平台覆盖。
