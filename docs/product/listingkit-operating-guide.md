# ListingKit 操作指南

## 目的

这份指南从用户任务出发说明如何使用 ListingKit，而不是按代码包解释实现。

它适用于运营培训、版本验收和产品需求评审。判断一个改动是否有效时，优先看它是否让下面的真实工作流更清楚、更稳定、更快完成。

## 主任务：创建并提交 SHEIN Listing

### 1. 选择正确入口

从 `/` 进入 ListingKit 首页，适合继续最近任务或选择工作区域。

从 `/listing-kits/shein` 进入 SHEIN Studio，适合从 SDS 选品和图片生成开始。

从 `/listing-kits/new` 创建通用 ListingKit 任务，适合从商品文本、图片 URL 或支持的商品 URL 开始。

### 2. 选择 SDS 商品上下文

在 SDS 商品库中，按发货地、分类、关键词和筛选条件找到目标商品，并选择要处理的变体。

SDS 变体会携带这些关键上下文：

- 商品和变体 ID
- 模板组
- 设计图层
- 可印刷区域
- mockup 和参考图
- 批量创建时的变体集合

如果这里选错，后续设计图和 SHEIN 任务可能看起来正常，但实际会落到错误商品或错误印刷区域。

### 3. 生成或导入设计图

在 SHEIN Studio 中，基于 SDS 上下文生成图片，或从图库导入已确认的图片。

创建任务前应确认：

- 图片内容符合目标商品和受众。
- 图片比例适配 SDS 可印刷区域。
- 被选中的图片就是要进入 ListingKit 任务的图片。
- 批量任务数量符合预期变体集合。

### 4. 创建 ListingKit 任务

图片审核通过后，创建 SHEIN ListingKit 任务。

创建成功后应能看到任务 ID，并进入任务跟踪或工作台审核。处于 `pending`、`queued`、`processing`、`running` 的任务应该继续显示处理状态，直到变成可审核或失败。

### 5. 审核工作台

打开 `/listing-kits/[taskId]/workspace`。

运营需要检查：

- 任务标题和状态。
- SHEIN 审核流程进度。
- 平台资料包摘要。
- 来源商品信息。
- 类目解析结果。
- 普通属性。
- 销售属性和变体映射。
- 图片图库和最终图。
- 价格、SKU、库存和数量。
- readiness 阻断项和提醒项。

readiness 面板是判断任务能否提交的主要依据。

### 6. 修复阻断项

出现阻断项时，优先点击阻断项动作，不要靠手动翻页面找问题。

当前已知修复目标：

- 类目阻断项进入类目审核。
- 必填属性阻断项进入普通属性。
- 变体映射阻断项进入销售属性。
- 图片阻断项进入图片资料和最终图。
- 价格、库存、stock、quantity 阻断项进入价格 / SKU 区域。

如果真实接口返回了无法跳转的新阻断项 key，先记录原始 key，再补映射，不能直接把这轮联调标记为通过。

### 7. 确认最终稿

提交前必须确认最终稿。这个动作应该发生在阻断项已修复、提醒项已理解之后。

最终稿确认的意义是：用户接受当前即将保存或发布的完整 SHEIN payload。因为工作台中可能存在中间态或刚修复的数据，所以最终确认不能省略。

### 8. 保存草稿或发布

选择 `保存到 SHEIN 草稿箱`，表示先在平台侧生成草稿，供后续人工检查。

选择 `发布到 SHEIN`，表示当前任务已经准备好通过远端 SHEIN 提交流程正式发布。

动作完成后，需要在当前工作台查看最新提交结果。失败时页面应该说明：

- 发生了什么。
- 影响是什么。
- 下一步应该做什么。

## 恢复处理

### 任务创建失败

先检查来源输入是否有效，以及错误信息是否能被运营理解。图片上传失败时，页面不应该只显示空响应或泛化失败。

下一步：

- 修复缺失的文本、图片 URL、商品 URL 或 SDS 选择。
- 重新创建任务。
- 如果 UI 无法解释失败，记录后端响应。

### 任务状态长时间不结束

确认任务是否仍处于合法非终态：

- `pending`
- `queued`
- `processing`
- `running`

如果超出预期处理时间仍无变化，应检查 child task 状态和后端日志。

下一步：

- 如果仍有明确处理进度，继续轮询。
- 如果状态为 `completed`、`succeeded`、`needs_review` 或 `review_ready`，进入工作台。
- 如果状态为 `failed` 或 `error`，停止自动跳转并展示失败恢复入口。

### 工作台无法加载

工作台依赖 preview、review session 和 task result。任意一类数据失败都可能导致页面无法完整加载。

下一步：

- 在页面点击重试。
- 检查 preview 是否存在。
- 检查 session 是否存在。
- 如果真实接口 payload 与前端契约不一致，记录缺失字段或新结构。

### readiness 阻断提交

不要绕过 readiness。按阻断项跳转到对应区域修复。

下一步：

- 修复映射到的工作台区域。
- 必要时保存 revision。
- 重新检查 readiness。
- 如果最终 payload 变化，重新确认最终稿。

### SHEIN 保存或发布失败

提交失败可能来自图片上传、payload 准备、预校验、SHEIN 远端校验或结果持久化。

下一步：

- 查看最新 submission report。
- 如果存在 validation notes，先按校验提示处理。
- 修复对应工作台区域。
- 等 UI 状态刷新后，由用户重新触发保存或发布。

## 版本验收清单

产品版本验收应至少用真实后端接口覆盖：

1. 从有效输入创建 ListingKit 任务。
2. 进入状态页，并看到预期非终态。
3. 在 `completed`、`succeeded`、`needs_review` 或 `review_ready` 后进入工作台。
4. 至少验证一个阻断项能跳到正确修复区域。
5. 保存一次 revision，并确认最终稿。
6. 保存到 SHEIN 草稿箱。
7. 发布到 SHEIN。
8. 主动触发并检查至少一个失败路径。
9. 确认没有未知任务状态。
10. 确认没有未知 readiness 阻断项 key。

字段级联调记录继续使用 `web/listingkit-ui/REAL_API_VALIDATION_CHECKLIST.md`。

## 运营可用标准

一个任务只有在用户能直接从 UI 回答这些问题时，才算达到运营可用：

- 我现在处在流程的哪一步？
- 当前任务是什么状态？
- 提交前必须修什么？
- 我点哪里修？
- 最终要提交的内容是什么？
- 平台动作是否成功？
- 如果失败，下一步应该做什么？

如果正常操作还需要读日志或找工程师解释，说明产品流程还没有闭环。
