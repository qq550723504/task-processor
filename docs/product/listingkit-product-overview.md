# ListingKit 产品总览

## 产品定位

ListingKit 是跨境商品上架的操作工作台，用来把来源商品素材转成可审核、可修复、可提交的平台 Listing 资料包。

在 `task-processor` 整体系统里，自动化 Worker 负责采集、结构化、图片生成、平台适配、校验和提交。ListingKit 负责把这些自动化结果产品化：让运营人员能看见结果、理解问题、修复阻断项，并把最终资料保存到平台草稿箱或正式发布。

当前产品重点是 SHEIN。Amazon、TEMU、Walmart 的资料包生成也在 ListingKit 目标平台范围内，但目前最完整的人工操作闭环是 SHEIN：SDS 选品、图片生成、ListingKit 任务创建、上架资料审核、最终稿确认、保存草稿或发布。

## 核心用户

### 上架运营

这是 ListingKit 的主要用户。用户需要创建或继续上架任务，检查自动化生成的资料，修复阻断项，并确认最终资料可以提交。

核心价值是提高产能并减少返工：少查日志、少反复刷新、少手动找问题，更快从选品进入平台草稿或发布。

### 运营负责人

运营负责人关注队列健康、失败任务、阻断类型和日常产出稳定性。他们需要知道哪些任务卡住了、哪些任务可以提交、哪些问题在重复出现。

核心价值是稳定产出和快速恢复：当图片、平台、店铺配置或数据异常时，能快速定位并推动恢复。

### 工程与 QA

工程和 QA 关注产品流程与接口契约是否一致。他们需要明确页面责任、状态语义、阻断项映射和真实接口验收标准。

核心价值是降低变更风险：改动后能确认没有破坏真实运营流程。

## 产品模块

### 首页

路由：`/`

首页是 ListingKit 的效率入口。它应该帮助用户快速进入最常用的工作区，而不是成为一个泛化大屏。

当前产品意图：

- 优先突出 SHEIN 工作流。
- 保留通用任务创建入口。
- 提供 SDS、任务列表、设置和最近任务入口。
- 让回访用户能尽快继续未完成任务。

### 任务创建

路由：

- `/listing-kits/new`
- `/listing-kits/shein`

任务创建把来源素材提交为异步 ListingKit 任务。

支持的来源素材：

- 商品图片 URL
- 商品描述文本
- 支持的商品 URL，当前重点是 1688 详情页
- SHEIN Studio 里带 SDS 上下文的选品和设计结果

任务创建页要在提交前暴露缺失输入；创建成功后进入任务状态页或任务跟踪视图。

### 任务状态

路由：`/listing-kits/[taskId]/status`

状态页展示后端异步生成进度。后端会执行商品理解、图片处理、平台资料包生成、校验等步骤。

状态页要回答：

- 任务是否已经开始？
- 任务是否仍在处理？
- 任务是否已经完成并可以审核？
- 任务是否失败，下一步应该怎么处理？

### 工作台

路由：`/listing-kits/[taskId]/workspace`

工作台是 ListingKit 的主审核和修复界面。

对 SHEIN 来说，工作台承载完整审核链路：

1. 查看商品事实和平台资料包。
2. 审核类目、普通属性、销售属性、图片、价格和 SKU。
3. 点击阻断项直接跳到对应修复区域。
4. 保存修改，必要时查看、对比或恢复历史版本。
5. 确认最终稿。
6. 保存到 SHEIN 草稿箱或正式发布。

### 队列

路由：`/listing-kits/[taskId]/queue`

队列页偏运营排查。它用于查看生成队列、恢复摘要、已解决动作、重试或检查入口，以及任务为什么仍然处于阻断状态。

### 任务列表

路由：`/listing-kits`

任务列表用于查找活跃、完成、失败和平台相关任务。它不只是浏览列表，更重要的是恢复和继续工作。

任务列表里的任务行应该把用户带到当前状态下最有用的继续位置，例如状态页、工作台或失败排查入口。

### SDS 商品库

路由：`/listing-kits/sds`

SDS 商品库用于在创建 SHEIN 设计和上架任务前选择底版商品和变体。

当前支持按发货地、分类、关键词、热卖、在售、重量、生产周期和排序条件查找商品。选中的 SDS 变体会把模板、印花层、可印刷尺寸、mockup 和参考图上下文带入 SHEIN Studio。

### SHEIN Studio

路由：

- `/listing-kits/shein`
- `/listing-kits/shein/batches/[batchId]`
- `/listing-kits/shein/gallery`
- `/listing-kits/style-gallery`

SHEIN Studio 负责基于 SDS 上下文生成或复用设计图，并把确认后的设计图转成 ListingKit 任务。

Studio 流程：

1. 选择 SDS 商品和变体。
2. 生成或导入设计图。
3. 审核图片结果。
4. 创建一个或多个 ListingKit 任务。
5. 跟踪任务处理状态。
6. 到工作台完成保存草稿或发布。

图库里的图片可以回流到 Studio，但最终提交仍然必须经过标准工作台和 readiness gate。

### 设置

路由：`/listing-kits/settings`

设置页管理会影响后续任务的客户级依赖，例如 AI 模型接口、SHEIN 店铺和价格规则。

这些配置不是个人偏好，而是运营控制项。错误配置会直接影响后续任务生成、图片处理或提交。

## 端到端产品流程

```text
选择来源素材或 SDS 商品
  -> 创建 ListingKit 任务
  -> 等待异步生成
  -> 进入工作台审核
  -> 修复阻断项
  -> 确认最终稿
  -> 保存 SHEIN 草稿或发布
  -> 查看提交结果
```

ListingKit 的产品承诺不是“生成一份资料”这么简单，而是提供一条可控路径：从原始商品素材到平台可用 Listing，中间每个自动化结果都能被理解、修复和确认。

## 产品状态模型

### 任务状态

任务状态描述后端异步任务：

- `pending` / `queued`：任务已接收但还未开始。
- `processing` / `running`：任务正在生成。
- `completed` / `succeeded`：生成结果可用。
- `needs_review` / `review_ready`：结果可用，但需要人工审核。
- `failed` / `error`：任务未成功完成。

### 审核状态

审核状态描述用户能否继续：

- 无阻断项：可以继续进入最终确认。
- 有提醒项：可以继续，但用户要理解风险。
- 有阻断项：必须先修复对应区域，不能提交。

### 提交状态

提交状态描述最终 SHEIN 动作：

- `pending_confirmation`：最终稿还未确认。
- `ready_to_submit`：允许保存草稿或发布。
- `draft_saved`：已保存到 SHEIN 草稿箱。
- `publish_failed`：发布失败，需要恢复。
- `published`：已发布。

## 产品原则

1. 下一步必须清楚。
   每个主要页面都要让用户知道当前状态和下一步动作。

2. 自动化结果必须可检查。
   商品事实、图片、属性和提交报告都要给足上下文，让用户能信任或修正。

3. 修复要能直达。
   阻断项应该把用户带到类目、属性、销售属性、图片、价格或 SKU 的具体修复区域。

4. 提交必须有门禁。
   保存草稿或发布前必须经过 readiness 检查和最终稿确认，避免把不完整资料发到平台。

5. 恢复能力是产品的一部分。
   图片生成失败、预览缺失、未知阻断项、SHEIN 远端校验失败，都应该有可执行的恢复路径。

- 产品级文档比部分工程文档更新，需要在路由或工作流责任变化时同步维护。
- SHEIN 当前拥有最完整的操作闭环；TEMU、Walmart 和 Amazon 的 ListingKit 产品流程还没有达到同等文档深度。
- 真实接口联调继续以 `web/listingkit-ui/REAL_API_VALIDATION_CHECKLIST.md` 为字段级记录模板。
- 队列页和任务列表的运营负责人视角还可以在后续补充更详细的 SOP。
