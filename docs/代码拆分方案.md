# Task Processor 代码拆分方案

生成时间: 2026-03-10

## 🎯 拆分目标

将当前的单体程序按功能拆分成独立的服务，每个服务可以独立部署、独立扩缩容。

## 📊 当前代码结构分析

### 当前入口程序

```
cmd/
├── rabbitmq-consumer/    # RabbitMQ 消费者（上架服务 + 爬虫服务）
├── task/                 # 定时调度任务
├── crawler-consumer/     # 爬虫消费者
└── test-*/              # 测试程序
```

### 当前问题

1. **单体架构**：所有功能在一个程序中
   - 上架服务（Amazon、TEMU、SHEIN）
   - 爬虫服务（Amazon、1688）
   - 运营服务（核价、库存、活动）

2. **无法独立扩展**：
   - 爬虫任务多时，无法只扩展爬虫服务
   - 上架任务多时，无法只扩展上架服务

3. **资源浪费**：
   - 所有服务共享资源
   - 无法针对性优化

## 🏗️ 拆分方案

### 方案概览

```
当前架构                    目标架构
┌─────────────────┐        ┌─────────────────┐
│ rabbitmq-       │        │ amazon-crawler  │
│ consumer        │   →    ├─────────────────┤
│                 │        │ 1688-crawler    │
│ - 所有平台      │        ├─────────────────┤
│ - 所有功能      │        │ temu-listing    │
│                 │        ├─────────────────┤
└─────────────────┘        │ shein-listing   │
                           ├─────────────────┤
                           │ pricing-service │
                           ├─────────────────┤
                           │ inventory-svc   │
                           └─────────────────┘
```

### 拆分后的服务

1. **amazon-crawler**：Amazon 爬虫服务
2. **1688-crawler**：1688 爬虫服务
3. **temu-listing**：TEMU 上架服务
4. **shein-listing**：SHEIN 上架服务
5. **amazon-listing**：Amazon 上架服务（如果需要）
6. **pricing-service**：自动核价服务
7. **inventory-service**：库存监测服务
8. **activity-service**：活动报名服务（未来）

### 步骤 3：修改代码支持服务拆分

#### 3.1 添加启动参数

每个服务通过启动参数指定功能：

```go
// cmd/amazon-crawler/main.go
package main

import (
    "flag"
    "task-processor/internal/app/messaging"
    "task-processor/internal/core/config"
)

var (
    configPath = flag.String("config", "config/config-prod.yaml", "配置文件路径")
    platform   = flag.String("platform", "amazon", "平台名称")
    mode       = flag.String("mode", "crawler", "运行模式: crawler, listing, pricing")
)

func main() {
    flag.Parse()
    
    // 加载配置
    cfg := config.LoadConfig(*configPath)
    
    // 根据 mode 启动不同的服务
    switch *mode {
    case "crawler":
        startCrawlerService(cfg, *platform)
    case "listing":
        startListingService(cfg, *platform)
    case "pricing":
        startPricingService(cfg, *platform)
    default:
        log.Fatalf("未知的运行模式: %s", *mode)
    }
}
```

## 💻 具体实现示例

### 示例 1：Amazon 爬虫服务

```go
// cmd/amazon-crawler/main.go
package main

import (
    "context"
    "flag"
    "os"
    "os/signal"
    "syscall"
    
    "task-processor/internal/app/messaging"
    "task-processor/internal/core/config"
    "task-processor/internal/pkg/utils"
)

var (
    configPath = flag.String("config", "config/config-prod.yaml", "配置文件路径")
    logLevel   = flag.String("log-level", "info", "日志级别")
)

func main() {
    flag.Parse()
    
    // 设置日志
    logger := utils.SetupLoggerWithLevel(*logLevel)
    logger.Info("🚀 启动 Amazon 爬虫服务...")
    
    // 加载配置
    cfg := config.LoadConfigWithFallback(*configPath, logger)
    
    // 创建服务管理器
    serviceManager, err := messaging.NewServiceManager(cfg.RabbitMQ, logger)
    if err != nil {
        logger.Fatalf("创建服务管理器失败: %v", err)
    }
    
    // 只注册 Amazon 爬虫处理器
    crawlerRegistry := messaging.NewCrawlerRegistry(cfg, logger, serviceManager.GetClient())
    if err := crawlerRegistry.RegisterAmazonCrawler(serviceManager); err != nil {
        logger.Fatalf("注册 Amazon 爬虫失败: %v", err)
    }
    
    // 启动服务
    ctx := context.Background()
    if err := serviceManager.Start(ctx); err != nil {
        logger.Fatalf("启动服务失败: %v", err)
    }
    
    logger.Info("✅ Amazon 爬虫服务启动完成")
    
    // 等待退出信号
    waitForShutdown(serviceManager, logger)
}

func waitForShutdown(sm *messaging.ServiceManager, logger *logrus.Logger) {
    sigChan := make(chan os.Signal, 1)
    signal.Notify(sigChan, os.Interrupt, syscall.SIGTERM)
    
    sig := <-sigChan
    logger.Infof("收到信号: %v，开始优雅关闭...", sig)
    
    sm.Stop()
    logger.Info("✅ 服务已优雅关闭")
}
```

### 3. 创建服务启动器

```go
// internal/app/bootstrap/service_starter.go
package bootstrap

type ServiceStarter struct {
    config *config.Config
    logger *logrus.Logger
}

func NewServiceStarter(cfg *config.Config, logger *logrus.Logger) *ServiceStarter {
    return &ServiceStarter{
        config: cfg,
        logger: logger,
    }
}

func (s *ServiceStarter) StartCrawlerService(platform string) error {
    serviceManager, err := messaging.NewServiceManager(s.config.RabbitMQ, s.logger)
    if err != nil {
        return err
    }
    
    crawlerRegistry := messaging.NewCrawlerRegistry(s.config, s.logger, serviceManager.GetClient())
    
    switch platform {
    case "amazon":
        if err := crawlerRegistry.RegisterAmazonCrawler(serviceManager); err != nil {
            return err
        }
    case "1688":
        if err := crawlerRegistry.Register1688Crawler(serviceManager); err != nil {
            return err
        }
    default:
        return fmt.Errorf("不支持的爬虫平台: %s", platform)
    }
    
    ctx := context.Background()
    return serviceManager.Start(ctx)
}

func (s *ServiceStarter) StartListingService(platform string) error {
    serviceManager, err := messaging.NewServiceManager(s.config.RabbitMQ, s.logger)
    if err != nil {
        return err
    }
    
    platformRegistry := messaging.NewPlatformRegistry(s.config, s.logger, platform)
    ctx := context.Background()
    
    switch platform {
    case "temu":
        if err := platformRegistry.RegisterTemuProcessor(ctx, serviceManager); err != nil {
            return err
        }
    case "shein":
        if err := platformRegistry.RegisterSheinProcessor(ctx, serviceManager); err != nil {
            return err
        }
    case "amazon":
        if err := platformRegistry.RegisterAmazonProcessor(ctx, serviceManager); err != nil {
            return err
        }
    default:
        return fmt.Errorf("不支持的上架平台: %s", platform)
    }
    
    return serviceManager.Start(ctx)
}
```

```makefile
# Makefile
.PHONY: all build-all clean
```

## 🎯 总结

### 拆分的好处

1. **独立扩展**：每个服务可以根据负载独立扩缩容
2. **故障隔离**：一个服务故障不影响其他服务
3. **技术灵活**：每个服务可以使用不同的技术栈
4. **团队协作**：不同团队可以并行开发
5. **资源优化**：按需分配资源，提高利用率

### 实施建议

1. **先重构，后容器化**：确保代码拆分正确
2. **逐步迁移**：一个服务一个服务地迁移
3. **保留备份**：保留原有程序作为备份
4. **充分测试**：每个阶段都要充分测试
5. **监控先行**：先完善监控，再做迁移

### 预期效果

- 代码更清晰，易维护
- 部署更灵活，易扩展
- 资源利用率更高
- 系统稳定性更好

这是一个**现代化、可扩展、易维护**的架构！
