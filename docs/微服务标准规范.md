# 微服务标准规范

## 概述

本文档定义了项目中微服务应该遵循的标准规范，确保服务的可观测性、可维护性和可扩展性。

## 一、必备功能

### 1.1 健康检查 (Health Check)

**目的**: 让容器编排系统（K8s）或负载均衡器判断服务是否存活。

**当前实现**: ✅ 已实现
- 端点: `GET /health`
- 端口: 8081
- 响应:
  ```json
  {
    "status": "healthy",
    "timestamp": "2024-03-10T10:00:00Z"
  }
  ```

**判断标准**:
- 服务进程正常运行
- 关键依赖可访问（RabbitMQ 连接正常）
- 系统资源未耗尽（CPU、内存在合理范围）

### 1.2 就绪检查 (Readiness Check)

**目的**: 判断服务是否准备好接收流量/任务。

**当前实现**: ✅ 已实现
- 端点: `GET /ready`
- 端口: 8081
- 响应:
  ```json
  {
    "ready": true,
    "timestamp": "2024-03-10T10:00:00Z"
  }
  ```

**判断标准**:
- RabbitMQ 连接已建立
- 必要的初始化已完成
- 依赖服务可访问

### 1.3 指标监控 (Metrics)

**目的**: 提供 Prometheus 格式的监控指标。

**当前实现**: ✅ 已实现
- 端点: `GET /metrics`
- 端口: 8082
- 格式: Prometheus 文本格式

**关键指标**:
- 任务处理数量（成功/失败）
- 任务处理耗时
- 队列消息数量
- 系统资源使用（CPU、内存）
- RabbitMQ 连接状态

### 1.4 统计信息 (Stats)

**目的**: 提供人类可读的统计信息。

**当前实现**: ✅ 已实现
- 端点: `GET /stats`
- 端口: 8082
- 格式: JSON

**包含信息**:
- 服务运行时长
- 任务处理统计
- 资源使用情况
- 错误统计

### 1.5 优雅关闭 (Graceful Shutdown)

**目的**: 确保服务关闭时不丢失正在处理的任务。

**当前实现**: ✅ 已实现
- 监听 SIGTERM、SIGINT 信号
- 停止接收新任务
- 等待当前任务完成
- 关闭连接和资源

## 四、是否需要添加 REST API？

### 4.1 当前架构分析

**当前模式**: 消息驱动（Message-Driven）
- 服务从 RabbitMQ 队列读取任务
- 处理完成后发送结果到队列
- 不需要外部直接调用

**优点**:
- 解耦：服务之间通过消息队列通信
- 异步：任务异步处理，不阻塞
- 可靠：消息队列保证消息不丢失
- 扩展：可以轻松增加消费者实例

**缺点**:
- 无法直接查询服务状态
- 无法手动触发特定操作
- 调试和测试相对困难

### 4.2 是否需要 REST API？

**场景一：纯后台处理服务**
- 如果服务只是从队列消费任务，不需要外部交互
- **结论**: 不需要 REST API
- **当前状态**: ✅ 符合要求

**场景二：需要管理和监控**
- 如果需要查询服务状态、暂停/恢复处理、调整配置
- **结论**: 需要简单的管理 API
- **建议**: 添加基础的管理端点

**场景三：需要手动触发任务**
- 如果需要通过 HTTP 请求触发任务（而不是只从队列读取）
- **结论**: 需要完整的 REST API
- **当前状态**: ❌ 不需要（任务由管理系统通过队列下发）

### 4.3 推荐方案

**最小化方案**（推荐）:
- 保持当前的健康检查和监控端点
- 不添加额外的 REST API
- 通过监控系统（Prometheus + Grafana）观察服务状态
- 通过日志系统（ELK/Loki）查看详细信息

**增强方案**（可选）:
- 添加简单的管理 API：
  - `GET /api/v1/info` - 服务信息
  - `GET /api/v1/stats` - 详细统计
  - `POST /api/v1/control/pause` - 暂停处理
  - `POST /api/v1/control/resume` - 恢复处理
- 用于运维管理，不用于业务逻辑
